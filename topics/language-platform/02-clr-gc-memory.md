## База: GC и память в .NET (Junior → Middle)

> Этот раздел — **цельная картина** до Senior-вопросов ниже. Если что-то непонятно в деталях — в чеклисте есть ссылка на углублённый разбор.

### 1. Зачем вообще GC

В C# ты пишешь `new User()` и **не вызываешь `free`**. Память под объект выделяет **CLR**, а освобождает **сборщик мусора (GC)** — когда объект **больше не нужен**.

| C / C++ | C# / .NET |
|---|---|
| `malloc` / `delete` — ты сам | `new` — CLR; освобождение — GC |
| Утечка, если забыл `delete` | «Логическая» утечка, если объект **достижим** ([q11](#q11)) |
| Детерминированное время освобождения | GC запускается **недетерминированно** |

**Managed (управляемая) память** — куча, которой владеет CLR. **Unmanaged** — память ОС, сокеты, файлы, нативные DLL: GC её **не закрывает** → `IDisposable` ([q6](#q6)).

---

### 2. Где что лежит: стек и куча

```
Поток
├── Стек — фреймы методов, локальные value-типы, ссылки (адреса на кучу)
└── ...

Процесс
└── Управляемая куча — объекты class, string, массивы, всё с new
```

- **Стек** — быстрый, автоматически очищается при выходе из метода. Переполнение → `StackOverflowException`.
- **Куча** — объекты **reference-типов**; живут, пока на них есть путь от **корня** ([q1](#q1), [q5](#q5)).
- Value-тип **внутри** объекта класса лежит **в куче** вместе с объектом, не «на стеке» ([тема 01](../01-csharp-language.md#q1)).

```csharp
void M()
{
    int a = 1;           // a — на стеке
    var user = new User(); // user (ссылка) — на стеке; объект User — в куче
}
```

---

### 3. Главный закон: достижимость (reachability)

GC **не** считает ссылки (`reference count`). Он спрашивает: **можно ли дойти до объекта от корня по цепочке ссылок?**

```
Корень (статика, стек, GCHandle…)
    → List<Order>
        → Order[]
            → Customer   ← ЖИВОЙ

Потерянная ссылка:
    temp = new byte[1_000_000];
    temp = null;   // массив НЕДОСТИЖИМ → кандидат на сборку
```

**Корни (roots):** локальные переменные в работающих методах, статические поля, CPU-регистры, `GCHandle`, очередь финализации ([q5](#q5)).

**Следствие:** объект «вам больше не нужен логически», но лежит в `static Dictionary` — для GC он **жив** → утечка.

---

### 4. Как GC собирает мусор: mark → sweep → compact

Кратко ([q4](#q4)):

1. **Mark** — от корней обойти граф, пометить достижимое.
2. **Sweep** — недостижимое считать мусором.
3. **Compact** — сдвинуть живые объекты, убрать «дыры», обновить ссылки.

На время части работы managed-потоки **останавливаются** (stop-the-world) → **GC pause** ([q13](#q13)).

**Pinned-объекты** ([q14](#q14)) нельзя сдвинуть → мешают compact → фрагментация.

---

### 5. Поколения: зачем Gen 0 / 1 / 2 и LOH

**Гипотеза:** большинство объектов умирают молодыми (`string` в цикле, DTO на запрос).

| Зона | Что там | Сборка |
|---|---|---|
| **Gen 0** | новые мелкие объекты | часто, дёшево |
| **Gen 1** | пережили Gen 0 | реже |
| **Gen 2** | долгоживущие (кэши, синглтоны) | редко, **дорого** (full GC) |
| **LOH** | объекты **≥ ~85 000 байт** | вместе с Gen 2; **по умолчанию не compact** → фрагментация ([q2](#q2)) |

**Мид-лайф кризис:** объект живёт «средне» (не секунды, не вечность) — доживает до Gen 2 и делает сборки тяжелее.

**SOH** (Small Object Heap) — Gen 0+1+2 для мелких объектов; в быту говорят просто «куча».

---

### 6. Когда запускается GC

Ты **не вызываешь** GC в нормальном коде. Триггеры (упрощённо):

- **Gen 0 переполнен** — слишком много новых аллокаций.
- **Давление на память** — пороги поколений, нехватка непрерывного блока.
- **Аллокация в LOH** — часто провоцирует Gen 2.
- **`GC.Collect()`** — принудительно; в проде **не лечение**, только диагностика.

Чем **больше аллокаций** в секунду → тем чаще GC → тем выше **GC pressure** (нагрузка на сборщик и паузы).

---

### 7. Что GC делает и чего НЕ делает

| GC освобождает | GC НЕ освобождает |
|---|---|
| managed-объекты в куче, когда недостижимы | файлы, сокеты, DB connection |
| память под `List`, `string`, DTO | нативные хэндлы за `FileStream` / `SqlConnection` |
| | соединение, пока объект жив и не `Dispose` |

```csharp
using var stream = File.OpenRead("a.txt");  // ✓ закрыть файл — твоя задача
var list = new List<int>();                 // ✓ память списка — GC
```

**Финализатор** (`~Class`) — **страховка** BCL: если забыли `Dispose`, GC **когда-нибудь** вызовет `~` и закроет native. Медленно, непредсказуемо ([q7](#q7)). В своём коде — `using`, не финализатор.

---

### 8. «Утечки памяти» в .NET

GC **не спасает**, если объект **достижим**:

- статическая коллекция без удаления;
- `event +=` без `-=`;
- кэш без TTL/лимита;
- таймер держит колбэк → держит `this`;
- незакрытый `IDisposable` (ресурс ОС + иногда долгоживущий объект).

Подробно — [q11](#q11).

---

### 9. WeakReference — «подсказка», а не удержание

**Обычная ссылка = сильная (strong):** пока есть хотя бы одна — GC **не соберёт** объект.

```csharp
byte[] big = new byte[10_000_000];
byte[] strong = big;   // две сильные ссылки на один массив
big = null;
// массив ЖИВ — strong ещё держит
```

**`WeakReference` = слабая ссылка:** объект **можно собрать**, даже если `WeakReference` ещё существует. `weak` **не продлевает жизнь** `bigObject` — это не счётчик и не «количество чего-то внутри».

```csharp
byte[] bigObject = LoadHugeCache();

var weak = new WeakReference(bigObject);  // weak — отдельный объект в куче
                                          // он только «знает адрес» bigObject

bigObject = null;   // сильных ссылок на массив больше нет

// --- GC может пройти в любой момент ---

if (weak.Target is byte[] stillAlive)
{
    // повезло: GC ещё не собрал — можно взять сильную ссылку из Target
    Use(stillAlive);
}
else
{
    // GC уже собрал массив — weak.Target == null
    bigObject = Reload();
}
```

**Кто сколько живёт:**

| Объект | Пока жив |
|---|---|
| `byte[] bigObject` | есть **сильная** ссылка (`bigObject`, поле, `Dictionary[key]`) |
| `WeakReference weak` | на `weak` кто-то ссылается (обычная сильная ссылка на сам `WeakReference`) |
| Связь `weak` → массив | **не** мешает GC собрать массив |

То есть: **`weak` может жить долго, а `bigObject` при этом уже мёртв** — `weak.Target` станет `null`.

**Зачем:** кэш «по возможности». «Если объект ещё в памяти — переиспользуй; если GC забрал — загрузи снова». Память не забита навсегя, в отличие от `Dictionary<string, byte[]>`.

```csharp
// ❌ сильный кэш — объект НИКОГДА не соберётся, пока ключ в словаре
_cache[key] = huge;

// ✓ мягкий кэш — GC может выкинуть huge при нехватке памяти
_weakCache[key] = new WeakReference(huge);
```

**Не путать:** `WeakReference` ≠ weak event pattern, ≠ `GCHandleType.Weak` (тот же смысл, другой API — [q14](#q14)).

---

### 10. Память процесса ≠ только managed-куча

В диспетчере задач **Working Set** процесса включает:

- managed-кучу (SOH, LOH, POH);
- **native** память (JIT-код, thread stacks, CLR itself, native DLL);
- **неуправляемые** буферы.

`dotnet-counters` → `gc-heap-size` — только **managed-куча**. Рост Working Set при стабильной куче → смотреть native, утечки хэндлов, LOH-фрагментацию ([q12](#q12)).

---

### 11. Практические привычки (Middle)

1. **`using` / `await using`** для всего `IDisposable` — файлы, `HttpResponseMessage`, DB.
2. **`IHttpClientFactory`**, не `new HttpClient()` на запрос ([q11](#q11)).
3. **Меньше аллокаций** на горячем пути: `StringBuilder`, generics вместо `ArrayList`, `Span`/`ArrayPool` ([q8](#q8), [q9](#q9)).
4. **Кэш с политикой** — TTL, max size; не `Dictionary` без границ.
5. **Отписываться от событий**, останавливать таймеры, не держать лишнее в `static`.
6. **Не вызывать `GC.Collect()`** в проде как «фикс».
7. Подозрение на утечку → **два снимка** кучи, не гадание ([q12](#q12)).

### 12. Мифы (часто путают на собесах)

| Миф | Реальность |
|---|---|
| «struct всегда на стеке» | поле класса — в куче; [тема 01](../01-csharp-language.md#q1) |
| «GC сразу удалит после `= null`» | удалит при **следующей** сборке, если объект недостижим |
| «В .NET утечек нет» | **логические** утечки есть ([q11](#q11)) |
| «`IDisposable` = память в куче» | `Dispose` = **ресурсы** (файл, сокет); кучу всё равно чистит GC |
| «Финализатор заменяет `using`» | финализатор — поздно и дорого ([q7](#q7)) |
| «Больше RAM = медленнее» | для GC важнее **частота аллокаций** и размер удерживаемого графа |

## Разбор вопросов

<a id="q1"></a>
### Вопрос: Стек vs управляемая куча — что где хранится
**Краткий ответ:**
- **Стек** — на поток, фиксированного размера (~1 МБ по умолчанию). Хранит фреймы вызовов: локальные value-переменные, параметры, адреса возврата. Выделение/освобождение — простое сдвигание указателя (быстро), память освобождается автоматически при выходе из метода.
- **Управляемая куча (managed heap)** — общая для процесса. Хранит объекты ссылочных типов и их содержимое. Управляется GC.
- Ссылочная переменная может лежать на стеке, а сам объект — в куче. Value-поля внутри объекта в куче лежат в куче.

**Подводные камни:**
- `StackOverflowException` (например, глубокая рекурсия) нельзя поймать `try/catch` — процесс падает.
- «Структуры всегда на стеке» — миф (см. тему 01).

<a id="q2"></a>
### Вопрос: Поколения GC, LOH, фрагментация
**Краткий ответ:**
- GC построен на гипотезе о поколениях: **большинство объектов живут недолго**.
- **Gen 0** — новые мелкие объекты; собирается чаще всего, дёшево. Пережившие переходят в Gen 1.
- **Gen 1** — буфер между кратко- и долгоживущими.
- **Gen 2** — долгоживущие объекты (синглтоны, кэши, статика); собирается редко, сборка дорогая (full GC).
- **LOH (Large Object Heap)** — объекты ≥ **85 000 байт** (большие массивы, буферы). Собирается вместе с Gen 2 и **по умолчанию не компактируется** → фрагментация. Можно принудительно: `GCSettings.LargeObjectHeapCompactionMode`.

**Подводные камни:**
- Частые большие аллокации (>85 КБ) фрагментируют LOH и вызывают дорогие full GC. Решение — `ArrayPool<T>`/`RecyclableMemoryStream`, переиспользование буферов.
- Мид-лайф кризис: объекты, живущие «средне», доживают до Gen 2 и удорожают сборку.

<a id="q3"></a>
### Вопрос: Workstation vs Server GC, Concurrent/Background GC
**Краткий ответ:**
- **Workstation GC** — оптимизирован под десктоп/низкую латентность, один heap (исторически). По умолчанию в клиентских приложениях.
- **Server GC** — несколько heap'ов и GC-потоков (по числу ядер), оптимизирован под пропускную способность серверов. По умолчанию в ASP.NET Core на многоядерных машинах. Больше памяти, выше throughput.
- **Background/Concurrent GC** — сборка Gen 2 выполняется в фоновом потоке, минимизируя паузы (stop-the-world).

**Настройка:** `<ServerGarbageCollection>true</ServerGarbageCollection>`, `<ConcurrentGarbageCollection>` в csproj или через env `DOTNET_gcServer`.

**Подводные камни:**
- В контейнерах с лимитами памяти Server GC может потреблять много RAM (heap на ядро). Учитывайте лимиты (`DOTNET_GCHeapHardLimit`, cgroup awareness).

<a id="q4"></a>
### Вопрос: Как работает mark-and-sweep + compaction
**Краткий ответ:**
1. **Mark** — обход графа объектов от корней (roots); достижимые помечаются как «живые».
2. **Sweep** — недостижимые считаются мусором, их память освобождается.
3. **Compact** — живые объекты сдвигаются вместе, устраняя «дыры»; обновляются ссылки. Это устраняет фрагментацию и делает аллокацию дешёвой (bump pointer). LOH по умолчанию не компактируется.

**Подводные камни:**
- Во время компактификации ссылки меняются — поэтому «pinned» объекты (см. ниже) мешают и фрагментируют кучу.

[↑ Наверх](#top)

<a id="q5"></a>
### Вопрос: Корни GC (roots)
**Краткий ответ:** Корни — точки входа, от которых GC считает объекты достижимыми:
- локальные переменные в активных фреймах стека и регистрах CPU;
- статические поля;
- объекты в очереди финализации;
- `GCHandle` (в т.ч. pinned).

Объект жив, пока достижим из любого корня.

**Подводные камни:**
- Статическое поле, ссылающееся на большой объект/коллекцию, держит его весь срок жизни приложения — классическая «утечка».

<a id="q6"></a>
### Вопрос: `IDisposable`, Dispose pattern, `using`, `IAsyncDisposable`

**Краткий ответ:**

- **GC** освобождает только **managed**-память. Файлы, сокеты, хэндлы ОС, нативная память (P/Invoke, Win32) — **вне GC** → нужен явный `Dispose`.
- **`IDisposable.Dispose()`** — **детерминированное** освобождение: вызываешь ты или `using` / `await using`.
- **Dispose pattern** — один приватный метод `Dispose(bool disposing)` на два входа: `true` из `Dispose()`, `false` из финализатора (см. [q7](#q7)).
- **Финализатор** в своих классах — редко; подробнее в [q7](#q7). В `Dispose()` после очистки — `GC.SuppressFinalize(this)`.

**Почему GC недостаточно:**

```csharp
var conn = new NpgsqlConnection(...);
// забыли Dispose — для GC объект «мусор», но TCP-соединение в ОС ещё живо
```

| Ресурс | Кто чистит без Dispose |
|---|---|
| `string`, `List<T>` | GC |
| файл, сокет, DB connection | **никто** (утечка до финализатора BCL или конца процесса) |

**`using` и `IAsyncDisposable`:**

```csharp
using var stream = File.OpenRead("data.bin");     // Dispose в конце scope, даже при exception

await using var conn = new NpgsqlConnection(cs);  // IAsyncDisposable → DisposeAsync
await conn.OpenAsync();
```

`using` — синтаксический сахар над `try/finally { dispose }`.

**Dispose pattern (полный):**

```csharp
public sealed class MyResource : IDisposable
{
    private IntPtr _nativeHandle;       // неуправляемое (P/Invoke, Win32)
    private Stream? _managedStream;   // управляемое
    private bool _disposed;

    public void Dispose()
    {
        Dispose(disposing: true);       // нормальный путь: ты / using
        GC.SuppressFinalize(this);      // финализатор больше не нужен
    }

    ~MyResource() => Dispose(disposing: false);  // страховка GC — см. q7

    private void Dispose(bool disposing)
    {
        if (_disposed) return;
        _disposed = true;

        if (disposing)
        {
            // disposing == true: из public Dispose(), мир живой
            _managedStream?.Dispose();
            _managedStream = null;
        }

        // и true, и false: закрыть нативное (GC не умеет)
        if (_nativeHandle != IntPtr.Zero)
        {
            NativeMethods.CloseHandle(_nativeHandle);
            _nativeHandle = IntPtr.Zero;
        }
    }
}
```

| `disposing` | Кто вызвал | Что можно чистить |
|---|---|---|
| `true` | `Dispose()` / `using` | managed **и** unmanaged |
| `false` | `~MyResource()` (GC) | **только** unmanaged |

**`~Class() => Dispose(false)`** — не «второй Dispose для ленивых», а **общий код** закрытия хэндла с ограничениями финализатора (не трогать другие managed-объекты).

**Когда финализатор в своём коде не нужен:**

- класс владеет **только managed** (`List`, сервисы, DTO) → достаточно `IDisposable` без `~Class`;
- нативное уже внутри BCL (`FileStream`, `SqlConnection`) → `using` на обёртке;
- современный вариант для native — **`SafeHandle`** (финализатор внутри BCL, ты его не пишешь).

**Типичное в приложении:**

```csharp
await using var db = await factory.CreateDbContextAsync();
await using var tx = await db.Database.BeginTransactionAsync();
// ...
```

**Подводные камни:**

- **Двойной `Dispose`** должен быть безопасен — флаг `_disposed` и guard в начале.
- Забытый `Dispose` у соединений → исчерпание портов/пула. Но **`HttpClient` из `IHttpClientFactory`** не оборачивать в `using` на каждый запрос (см. тему 11).
- Финализатор без `GC.SuppressFinalize` в `Dispose` → объект **лишний раз** переживает GC (дорого).
- `Dispose` освобождает **ресурсы**, не «уничтожает объект в памяти» — сам managed-объект соберёт GC, когда на него не останется ссылок.

<a id="q7"></a>
### Вопрос: Финализация, очередь финализации, `GC.SuppressFinalize`

**Краткий ответ:**

- **`~Class()`** — **не** деконструктор C++. Это **финализатор**: его вызывает **только GC** (поток финализатора), **не** ты и **не** `using`.
- **Зачем:** страховка, если забыли `Dispose`, а класс владеет **неуправляемым** ресурсом (Win32 handle, P/Invoke). GC сам файлы/сокеты **не** закрывает.
- При `new` объект с `~Class` регистрируется в **очереди финализации**. Когда ссылок нет → попадает в **f-reachable queue** → финализатор → только потом объект собирается. **Минимум один лишний цикл GC.**
- **`GC.SuppressFinalize(this)`** в `Dispose()` — снять с очереди, если уже освободили ресурсы детерминированно (см. [Dispose pattern в q6](#q6)).

**`Dispose` vs финализатор:**

| | `Dispose()` | `~Class()` |
|---|---|---|
| Кто вызывает | разработчик, `using` | поток финализатора GC |
| Когда | сразу, детерминированно | когда-нибудь |
| Вручную | да | **нет** (только `SuppressFinalize` отменяет) |
| Managed-ресурсы | можно | **нельзя** (могут быть уже собраны) |
| Назначение | основной путь | last resort для native |

**Очередь финализации (упрощённо):**

```
new MyResource()  →  в очередь финализации
ссылок нет        →  объект «мусор», но ещё не удалён
GC                →  перенос в f-reachable, вызов ~MyResource() → Dispose(false)
следующий GC      →  память managed-обёртки
```

**Связь с Dispose pattern:**

```csharp
public void Dispose()
{
    Dispose(true);
    GC.SuppressFinalize(this);   // «мы уже всё закрыли — финализатор не нужен»
}

~MyResource() => Dispose(false);  // только native, без _managedStream?.Dispose()
```

**Почему финализатор хрупкий:**

- **недетерминирован** — до срабатывания порты/файлы могут кончиться;
- **порядок** финализации не гарантирован;
- **нельзя** полагаться на другие managed-объекты в `~Class`;
- **исключение** в финализаторе → краш процесса.

**Почему не «сделаю финализатор — можно забить на Dispose»:**

- все экземпляры с `~Class` дороже для GC, даже если все вызывают `Dispose`;
- это **не** гарантия «ничего не сломается», а отсроченная уборка native.

**Современная альтернатива:** `SafeHandle` — нативный хэндл + финализация внутри BCL; в прикладном коде — `using` на `FileStream` / `SqlConnection`.

**Подводные камни:**

- Финализатор в классе **без** прямого native — лишняя нагрузка на GC, не добавляй «на всякий случай».
- Без `SuppressFinalize` в `Dispose` — двойная работа и лишний GC-цикл.
- Путать с `IDisposable` из BCL: `FileStream` уже всё сделал — тебе нужен только `using`.

<a id="q8"></a>
### Вопрос: Boxing/unboxing — где аллокации, как избежать

**Краткий ответ:**

- **Boxing** — value-тип (`int`, `struct`, `enum`) **упаковывается** в объект на **куче**: аллокация + копия значения. Переменная `object`/`interface` хранит **ссылку** на этот объект.
- **Unboxing** — извлечение value обратно: `(int)o` + **проверка типа** в рантайме.
- Скрытый боксинг — частый источник мусора в hot path и нагрузки на GC. Подробнее value vs reference — [тема 01, q1](./01-csharp-language.md#q1).

**Что происходит в памяти (boxing):**

```csharp
int x = 42;        // 4 байта на стеке (или внутри объекта)
object o = x;      // boxing
```

```
До:   x → [ 42 ]                    (value на стеке)

После boxing:
      x → [ 42 ]                    (оригинал остаётся)
      o → ●──► [ object в куче: type=System.Int32, value=42 ]
               ↑ новая аллокация + копия
```

`object` может ссылаться только на **объект в куче**. `int` — не объект → CLR создаёт **обёртку** `System.Int32` в куче и копирует туда `42`.

**Unboxing:**

```csharp
int back = (int)o;   // unboxing: проверка «это Int32?» → копия 42 обратно в back
```

Не путать с **cast** ссылочных типов — unboxing только для value, с проверкой типа (`InvalidCastException`).

**Разбор примеров построчно:**

```csharp
int x = 5;
object o = x;
```

Явный boxing: `int` → `object`. Видно в коде.

```csharp
IComparable c = x;
```

**Скрытый boxing.** `IComparable` — **интерфейс** (ссылочный тип). `int` реализует `IComparable`, но **на стеке** интерфейс не «лежит» — CLR боксит `x` в объект в куче и кладёт ссылку в `c`.

```csharp
Console.WriteLine($"{x}");
```

Зависит от версии/перегрузки:
- старый `string.Format("...", x)` — `x` в `object[]` → **boxing**;
- современный `$"{x}"` с `DefaultInterpolatedStringHandler` — часто **без** boxing для `int` (специализированные перегрузки).

**Другие частые случаи:**

```csharp
// Не-generic коллекции — каждый Add боксит
var al = new ArrayList();
al.Add(42);                    // boxing: int → object

// Generic — без boxing
var list = new List<int>();
list.Add(42);                   // OK: T = int

// enum → object / не-generic
DayOfWeek d = DayOfWeek.Monday;
object od = d;                  // boxing
al.Add(d);                      // boxing

// Dictionary с value-типом без IEquatable — сравнение может боксить
struct Point { public int X, Y; }
var dict = new Dictionary<Point, string>();  // лучше Equals/GetHashCode + IEquatable<Point>
```

**Почему это важно для GC (тема 02):**

Каждый boxing = **аллокация в Gen 0**. В цикле на 100 000 итераций:

```csharp
// плохо в hot path
for (int i = 0; i < 100_000; i++)
    log.Add(i);   // ArrayList.Add → 100k boxing → 100k объектов в куче

// хорошо
for (int i = 0; i < 100_000; i++)
    list.Add(i);  // List<int> — без boxing
```

**Как избежать:**

| Проблема | Решение |
|---|---|
| `object` / интерфейс на value | дженерики `List<T>`, `Dictionary<TKey,TValue>` |
| `ArrayList`, `Hashtable` | `List<T>`, `Dictionary<,>` |
| сравнение struct в `Dictionary` | `IEquatable<T>` + переопределить `Equals`/`GetHashCode` |
| логирование value | structured logging: `_log.LogInfo("x={X}", x)` — часто без boxing |
| enum в коллекции | `List<MyEnum>`, не `ArrayList` |
| `string.Format` с value | интерполяция / `CompositeFormat` / перегрузки BCL |

**Мини-чеклист «есть ли boxing»:**

```
value-тип присваивают object / interface?     → да
кладут в ArrayList / Hashtable?               → да
вызывают не-generic API с value-аргументом?   → часто да
List<int> / Dictionary<string,int>?            → нет
```

**Подводные камни:**

- Boxing **не виден** в IDE — компилятор не предупреждает.
- Unboxing **копирует** value — мутация после unboxing не меняет объект в куче.
- `nullable value` (`int?`) при boxing в `object` — отдельные правила; `null` → null-ссылка без boxing значения.
- Микробенч без профайлера легко врёт; смотреть **alloc rate** (`dotnet-counters`).

<a id="q9"></a>
### Вопрос: `Span<T>`, `Memory<T>`, `stackalloc`, `ArrayPool<T>`

**Краткий ответ:**

- **`Span<T>`** / **`ReadOnlySpan<T>`** — срез (view) на **непрерывную** память **без копии**: массив в куче, `stackalloc`, native. `ref struct` — нельзя в полях класса и через `await` (см. тему 01).
- **`Memory<T>`** — тот же смысл «окна», но **можно** в поля и `async`; синхронная работа — через `.Span`.
- **`stackalloc`** — буфер **на стеке**, GC не участвует; только локально и **небольшой** размер.
- **`ArrayPool<T>`** — переиспользование **больших** `T[]` из кучи без `new` на каждый запрос; обязательно `Return`.

**Когда что:**

| Задача | Инструмент |
|---|---|
| Срез `string`/`byte[]` без `Substring` / копии | `str.AsSpan()`, `buf.AsSpan(offset, len)` |
| Синхронный парсинг / валидация | `Span` / `ReadOnlySpan` |
| `await ReadAsync` / поле класса | `Memory<T>` или `byte[]` |
| Маленький temp-буфер в методе | `stackalloc` |
| Большой буфер на многократное использование | `ArrayPool<T>.Shared` |

**`Span` vs `Memory`:**

```csharp
byte[] buf = ArrayPool<byte>.Shared.Rent(4096);

// sync — Span (или просто buf)
Span<byte> slice = buf.AsSpan(0, read);
ParseHeader(slice);              // изменения в slice → видны в buf

// async — Memory (Span в await нельзя)
await stream.ReadAsync(buf.AsMemory(0, buf.Length));
Span<byte> after = buf.AsSpan(0, read);
```

`Memory` не «дороже» — его берут, когда **`Span` запрещён** (`async`, поля). В sync предпочитай `Span` или `byte[]` напрямую.

**`new byte[n]` vs `stackalloc`:**

```csharp
byte[] heap = new byte[100];           // byte[] — reference type, данные в КУЧЕ, GC
Span<byte> stack = stackalloc byte[100]; // байты на СТЕКЕ, без GC, только в методе
```

`new byte[]` на стеке лежит только **ссылка**; сами байты — в куче.

**Срез без лишней аллокации:**

```csharp
// плохо — новая строка в куче
string part = json.Substring(10, 4);

// хорошо — view на тот же string
ReadOnlySpan<char> part = json.AsSpan(10, 4);
ValidateCode(part);
```

Для HTTP/high-perf: валидация фиксированных полей, парсинг по `ReadOnlySpan<char>` или UTF-8 `ReadOnlySpan<byte>` без промежуточных `string`.

**`ArrayPool<T>` — логика и жизненный цикл:**

`ArrayPool<T>.Shared` — глобальный **склад массивов** (`System.Buffers`), не «runtime сам подсовывает». Ты явно **берёшь** и **возвращаешь**:

```csharp
var buf = ArrayPool<byte>.Shared.Rent(4096);  // взять
try { ... }
finally { ArrayPool<byte>.Shared.Return(buf); } // вернуть на склад
```

**Первый `Rent` → аллокация; следующий часто без `new`:**

```
Rent (пул пуст)     →  new byte[4096] в куче
Return              →  массив в пуле (GC его НЕ собирает — пул держит ссылку)
Rent снова          →  тот же/похожий массив из пула, без new ✅
```

`Return` — **не** «освободить память ОС», а **вернуть на склад** внутри процесса для hot path (HTTP, TCP, чтение потоков).

**Размер и корзины (buckets):**

```csharp
byte[] buf = ArrayPool<byte>.Shared.Rent(4000);
// buf.Length часто 4096 — ≥ запрошенного, не обязательно ровно
Process(buf.AsSpan(0, read));   // работай только с нужным префиксом
```

Пул группирует массивы по размерам (часто степени двойки). Другой `Rent(100_000)` — другая корзина, может снова понадобиться `new`.

**`ArrayPool` vs `stackalloc` (смысл, не синтаксис):**

| | `stackalloc` | `ArrayPool` |
|---|---|---|
| Где данные | стек | куча (но **переиспользуется**) |
| GC при работе | нет | массив в куче; меньше **новых** `new` |
| Размер | мало (байты–КБ) | КБ–МБ (в т.ч. LOH) |
| Жизнь | до конца метода | от `Rent` до `Return` |
| Кейс | temp в парсинге | буфер на `ReadAsync` / каждый запрос |

Один `Rent` — несколько логических зон **без** лишних `new[]`:

```csharp
Span<byte> header = buf.AsSpan(0, 128);
Span<byte> body    = buf.AsSpan(128, read - 128);
```

**`ArrayPool` + `stackalloc` вместе:**

```csharp
byte[] buf = ArrayPool<byte>.Shared.Rent(8192);
try
{
    int n = await stream.ReadAsync(buf.AsMemory(0, buf.Length));
    Span<char> id = stackalloc char[16];   // мелочь — стек
    ExtractId(buf.AsSpan(0, n), id);
}
finally { ArrayPool<byte>.Shared.Return(buf); }
```

**Типичный паттерн:**

```csharp
byte[] buffer = ArrayPool<byte>.Shared.Rent(4096);
try
{
  int read = await stream.ReadAsync(buffer.AsMemory(0, buffer.Length));
  Process(buffer.AsSpan(0, read));
}
finally
{
  ArrayPool<byte>.Shared.Return(buffer);  // после Return массивом не пользоваться
}
```

**Подводные камни:**

- `stackalloc` большого размера → `StackOverflowException`; ориентир — сотни байт / единицы КБ, не МБ.
- `Span` на `ArrayPool`-буфер **нельзя** хранить в поле класса дольше `Return` — другой поток может получить тот же массив (use-after-free).
- Забытый `Return` → массив **не** возвращается в пул → снова `new` на каждый запрос + рост памяти процесса.
- Пул **не очищает** массив после `Return` — следующий `Rent` может увидеть старые байты; при необходимости `buf.AsSpan(..).Clear()`.
- Корзина пула **не бесконечная** — лишние `Return` могут не сохраниться; тогда массив станет обычным мусором для GC.
- `AsSpan` **мутирует** исходный массив/`string` (для `string` только `ReadOnlySpan`).
- `Memory` «везде вместо Span» — лишний слой; многие API принимают только `Span` (`TryParse`, `stackalloc`).

<a id="q10"></a>
### Вопрос: JIT, AOT, Tiered Compilation, ReadyToRun

**Откуда берутся термины — цепочка выполнения .NET:**

```
C#  →  компилятор (Roslyn)  →  IL в .dll (промежуточный язык, не машинный код)
                                      ↓
                            CLR при запуске / первом вызове
                                      ↓
                            нативный код CPU (x64, ARM…)
```

**IL (Intermediate Language)** — байткод для виртуальной машины CLR, как «универсальный чертёж». **CPU его не выполняет напрямую** — нужен перевод в машинные инструкции.

---

**JIT (Just-In-Time) — «компиляция по требованию»**

**Just-In-Time** = **в последний момент**, прямо **перед/при первом вызове** метода.

```csharp
static int Add(int a, int b) => a + b;

Add(1, 2);  // ← первый вызов: CLR/JIT компилирует Add в машинный код
Add(3, 4);  // уже выполняется готовый нативный код
```

| Плюс | Минус |
|---|---|
| быстрый **старт** приложения (не всё компилируется сразу) | **пауза** при первом вызове «холодного» метода |
| оптимизации под **конкретный CPU** | нужен **runtime** (CLR) на машине |
| можно загружать сборки динамически | |

**По умолчанию** обычный `dotnet run` / ASP.NET Core — **JIT**.

---

**AOT (Ahead-Of-Time) — «скомпилировали заранее»**

**Ahead-Of-Time** = **до запуска** (при сборке/публикации), а не в рантайме.

Общая идея: IL → нативный код **заранее**. В .NET это **не один режим**, а спектр:

| Технология | Когда компилируют | JIT ещё нужен? |
|---|---|---|
| **ReadyToRun (R2R)** | `dotnet publish` | частично да |
| **Native AOT** | `dotnet publish` | **нет** (почти) |

---

**Tiered Compilation — «двухэтажный JIT»**

Проблема чистого JIT: полная оптимизация **долго** → холодный старт API тормозит.

**Решение (.NET Core 3+):** компилировать в **уровни (tiers)**:

```
Tier 0  —  быстрая, слабо оптимизированная версия метода (быстрый первый вызов)
   ↓  метод часто вызывается («горячий»)
Tier 1  —  пере-компиляция с полными оптимизациями
```

Аналогия: сначала **черновик**, потом **финальная версия** для горячих мест.

**Tiered PGO** (Profile-Guided Optimization) — Tier 1 с учётом **реальной статистики** вызовов (какие ветки `if` чаще). Включение: `DOTNET_TieredPGO=1` или в csproj.

**По умолчанию** Tiered Compilation в современных .NET **включён**.

---

**ReadyToRun (R2R) / Crossgen2 — гибрид**

При `dotnet publish` часть IL **предкомпилируется** в нативный код и кладётся в сборку рядом с IL.

```
Старт приложения:
  есть R2R для метода  →  быстрее, меньше работы JIT
  нет R2R              →  обычный JIT как раньше
```

| Плюс | Минус |
|---|---|
| быстрее **cold start** (контейнеры, serverless) | **больше** размер publish |
| JIT остаётся для остального | R2R привязан к **целевой OS/CPU** при публикации |

Включается, например: `<PublishReadyToRun>true</PublishReadyToRun>`.

Это **не полный AOT** — рефлексия, `Assembly.Load`, динамика в основном работают.

---

**Native AOT — полная нативная сборка**

`PublishAot` — IL **целиком** (плюс runtime) компилируется в **один нативный** exe / библиотеку **без** классического JIT в рантайме.

| Плюс | Минус |
|---|---|
| очень быстрый старт | нет полноценной **рефлексии** / динамической загрузки |
| меньше RAM, меньше поверхность атаки | нужен **trimming** — лишний код вырезается |
| удобно для CLI-утилит, контейнеров | EF, reflection-heavy библиотеки часто **ломаются** без доработки |

Для типичного ASP.NET с EF — **не** дефолт; для микро-сервисов, tools, workers — рассматривают осознанно. Нужны **source generators** вместо рефлексии (см. тему 01, q17).

---

**Почему Native AOT «ломает» рефлексию?**

Частая путаница: «код же зафиксирован, все имена классов известны — почему `Type.GetType` не работает?»

Ломает не столько **сама AOT-компиляция**, сколько **trimming (обрезка)** при `publish`, которая идёт вместе с Native AOT:

| Этап | Кто | Что делает |
|---|---|---|
| `dotnet build` | Roslyn | C# → IL, всё на месте |
| `dotnet publish` (Native AOT) | **trimmer / линкер** | убирает код без **явных ссылок** в IL |

Линкер строит граф: «на этот тип/конструктор/метод кто-то **напрямую** ссылается?» Если нет — считает **мёртвым кодом** и вырезает (иногда **весь тип** целиком, не только конструктор).

```csharp
// Явная ссылка — линкер видит: MyHandler нужен ✓
var h = new MyHandler();

// Рефлексия — связи в графе нет ✗
var type = Type.GetType("MyApp.MyHandler");
var obj = Activator.CreateInstance(type);  // в publish типа уже нет
```

Для **человека** имя `"MyApp.MyHandler"` известно. Для **статического анализатора** строка — не ссылка → тип вырезан → в рантайме рефлексия не находит метаданные.

**Что ещё страдает кроме конструкторов:**

- `assembly.GetTypes()` — сканирование сборки;
- EF / старый JSON — обход свойств через reflection;
- `Assembly.LoadFrom("plugin.dll")` — динамическая загрузка;
- `Reflection.Emit`, `DynamicMethod` — нужен JIT в рантайме, в Native AOT недоступен.

**В обычном JIT без trimming** всё из `.dll` остаётся — рефлексия работает. Проблема именно связка **Native AOT + агрессивная обрезка**.

**Как сохранить нужное:**

```csharp
// Прямая ссылка — линкер не вырежет
typeof(MyHandler).GetMethod(nameof(MyHandler.Run));

// Или явная пометка для trimmer
[DynamicallyAccessedMembers(DynamicallyAccessedMemberTypes.PublicConstructors)]
public class MyHandler { ... }
```

**Source generators** — compile-time замена reflection: компилятор **вшивает** знание о типах в IL, линкер видит прямые ссылки → ничего не режет (см. тему 01, q17).

---

**Сводная таблица «что спросят на собесе»:**

| Термин | Расшифровка | Суть одной фразой |
|---|---|---|
| **JIT** | Just-In-Time | IL → машинный код **при первом вызове** |
| **AOT** | Ahead-Of-Time | машинный код **до** запуска |
| **Tiered Compilation** | многоуровневый JIT | сначала быстрый код, потом оптимизированный для hot path |
| **ReadyToRun** | R2R, crossgen | **частичный** AOT при publish, JIT остаётся |
| **Native AOT** | полный AOT | нативный бинарник, без JIT, жёсткие ограничения |

**Что у тебя в жизни:** обычный ASP.NET / консоль — **JIT + Tiered Compilation**. R2R / Native AOT — осознанный выбор под **cold start** и размер, не «всегда лучше».

**Подводные камни:**

- Путать **R2R** и **Native AOT**: R2R — гибрид с JIT; Native AOT — другой мир с trimming.
- «Первый запрос медленный» в API — часто **cold JIT** + инициализация EF, не только GC.
- Native AOT ломает рефлексию в основном из‑за **trimming**: нет явных ссылок на тип/конструктор → линкер вырезал → `Activator.CreateInstance` падает.
- Native AOT + `dynamic` / загрузка плагинов / `Reflection.Emit` — отдельные ограничения (нет runtime JIT).
- R2R увеличивает артефакт publish — плохо для частых мелких деплоев без выигрыша по старту.

<a id="q11"></a>
### Вопрос: Утечки памяти в .NET — причины

**Главное:** GC убирает только **недостижимые** managed-объекты. «Утечка» в .NET — объект (или ресурс) **всё ещё достижим** из корней, хотя логически он уже не нужен.

**Уже разобранные паттерны (кратко):**

| Причина | Суть |
|---|---|
| **Статические коллекции/поля** | `static List<>` / `Dictionary` живёт весь процесс → всё, что туда `Add`, **никогда** не соберёт GC |
| **События `+=` без `-=`** | паблишер хранит делегат → делегат держит подписчика (`this`) |
| **Замыкания** | лямбда захватила `this` или большой объект → попала в долгоживущий таймер/кэш/статику |

Ниже — три причины, которые часто путают с «GC всё почистит».

---

**1. `IDisposable`, который не диспоузят**

Подробнее про паттерн — [q6](#q6). Здесь — **почему это утечка**, а не «GC сам разберётся».

GC видит **managed-обёртку** (`FileStream`, `SqlConnection`, `HttpResponseMessage`). Нативное за ней — **сокет, файловый дескриптор, буфер в ОС** — GC **не закрывает**.

```csharp
void Bad()
{
    var conn = new NpgsqlConnection(cs);
    conn.Open();
    // забыли using / Dispose — метод завершился
}
// Ссылка на conn пропала → для GC объект «мусор»
// Но TCP-соединение в PostgreSQL ещё открыто (секунды/минуты до финализатора BCL)
```

Два вида вреда:

| Вид | Что происходит |
|---|---|
| **Утечка ресурса ОС** | сокеты, файлы, соединения копятся → `too many open files`, пул соединений исчерпан |
| **Утечка managed-памяти** | если **кто-то ещё** держит ссылку на `IDisposable` и не вызывает `Dispose` — объект + его буферы остаются в куче |

Типичные места — **проблема не в том, что `using` плохой**, а в том, что его **забывают** или применяют не к тому объекту:

```csharp
// --- HttpClient: отдельный случай ---

// ❌ new HttpClient() на каждый запрос — socket exhaustion
//    (даже с using на каждый раз — антипаттерн: убивается connection pool handler'а)
var client = new HttpClient();
var json = await client.GetStringAsync(url);
client.Dispose();

// ✓ IHttpClientFactory из DI — CreateClient() без using
//    (экземпляр короткоживущий, handler'ы живут в фабрике; Dispose у такого client — no-op)
var client = _httpClientFactory.CreateClient();
var json = await client.GetStringAsync(url);

// --- FileStream: классический using ---

// ❌ без using — файловый дескриптор висит до финализатора
var stream = File.OpenRead(path);

// ✓
using var stream = File.OpenRead(path);

// --- HttpResponseMessage: вот тут using обязателен ---

// ❌ response не disposed — тело/соединение не освобождено, пул сокетов забивается
var response = await client.GetAsync(url);
var json = await response.Content.ReadAsStringAsync();

// ✓ disposing response (достаточно обычного using; await using тоже ок)
using var response = await client.GetAsync(url);
var json = await response.Content.ReadAsStringAsync();
```

| Объект | Нужен `using`? | Почему |
|---|---|---|
| `HttpClient` из **фабрики** | **нет** на каждый `CreateClient()` | фабрика держит handler'ы; см. тему 11 |
| `new HttpClient()` | dispose нужен, но **лучше не создавать** так | антипаттерн |
| `FileStream`, `SqlConnection` | **да** | нативный хэндл |
| `HttpResponseMessage` | **да** | вернуть соединение в пул |

**На собесе:** GC освобождает **память в куче**, не **внешние ресурсы**. `using` / `await using` — не «стиль», а предотвращение утечки хэндлов.

---

**2. Кэши без вытеснения**

Кэш — **намеренно** долгоживущая структура. GC не трогает записи, пока на них есть ссылка из кэша.

```csharp
// Singleton-сервис на всё время приложения
private readonly Dictionary<int, byte[]> _cache = new();

public byte[] GetReport(int userId)
{
    if (!_cache.TryGetValue(userId, out var data))
    {
        data = BuildHugeReport(userId);  // мегабайты
        _cache[userId] = data;           // навсегда
    }
    return data;
}
```

Каждый новый `userId` → новая запись → **куча растёт монотонно**. Это не баг GC, это **логическая** утечка: данные «забыли», но кэш их держит.

Признаки:

- память растёт **пропорционально** числу уникальных ключей / запросов;
- в дампе — огромный `Dictionary<>` / `ConcurrentDictionary` / `MemoryCache` внутри singleton.

**Как правильно:**

```csharp
// IMemoryCache с лимитом и TTL
services.AddMemoryCache(o => o.SizeLimit = 10_000);

_cache.Set(key, value, new MemoryCacheEntryOptions
{
    Size = 1,
    SlidingExpiration = TimeSpan.FromMinutes(10),
    AbsoluteExpirationRelativeToNow = TimeSpan.FromHours(1)
});

// или LRU / вытеснение по размеру — библиотеки, свой ring-buffer
```

**Правило:** у любого кэша должен быть ответ на «что и **когда** выкидываем?» — TTL, LRU, max size, scoped lifetime.

---

**3. Таймеры / `Timer`, удерживающие колбэк**

`System.Threading.Timer` (и похожие) хранят **сильную ссылку** на делегат колбэка. Делегат держит **цель** (`this` для instance-метода) → весь объект и всё, что он захватил, **живут**, пока таймер не остановлен.

```csharp
public class SyncJob
{
    private readonly List<ProjectDto> _buffer = new();  // большой список

    public void Start()
    {
        var timer = new Timer(_ => DoSync(), null,
            TimeSpan.Zero, TimeSpan.FromMinutes(1));
        // timer нигде не сохранили и не Dispose — но он УЖЕ зарегистрирован в ThreadPool
        // колбэк = замыкание/метод → держит SyncJob (или захваченные поля)
    }
}
```

Даже если локальная переменная `timer` «пропала» — **планировщик** всё равно держит таймер и колбэк. `SyncJob` не соберётся, `_buffer` тоже.

Связь с замыканиями (п. 3 в таблице выше): колбэк таймера часто **лямбда**, которая захватила `this` или большой объект.

```csharp
// ❌ лямбда захватила hugeList
var timer = new Timer(_ => Process(hugeList), ...);

// ✓ остановить и освободить
_timer?.Dispose();
_timer = null;

// ✓ в ASP.NET Core — IHostedService + CancellationToken вместо «голого» Timer
```

| API | Что забывают |
|---|---|
| `System.Threading.Timer` | `Dispose()` |
| `System.Timers.Timer` | `Stop()` + `Dispose()` |
| `PeriodicTimer` | `Dispose()` + отмена `CancellationToken` |

**Диагностика:** снять два дампа памяти (`dotnet-gcdump` / dotMemory), сравнить рост объектов, найти **retention path** — цепочку ссылок от корня до «застрявшего» объекта.

**Подводные камни:**

- «Объект уже не в переменной» ≠ «GC может собрать» — статика, события, кэш, таймер держат сильные ссылки.
- Незакрытый `IDisposable` — утечка **ОС-ресурсов** быстрее, чем рост managed-кучи.
- Кэш без политики вытеснения в long-running API (weeksync, sync jobs) — частая причина роста памяти «со временем».
- `GC.Collect()` не лечит логические утечки — объект всё ещё достижим.

### Вопрос: Инструменты диагностики

**Как запомнить:** не «зоопарк ради зоопарка», а **три задачи** — смотреть **сейчас**, снять **снимок кучи**, записать **что происходило во времени**.

```
«Память растёт?»  →  dotnet-counters          (живые цифры)
«Кто держит объекты?»  →  dotnet-gcdump × 2    (сравнить снимки)
«Почему тормозит / много GC?»  →  dotnet-trace / PerfView
«Прод упал / завис»  →  dotnet-dump            (полный дамп на потом)
```

Все `dotnet-*` — **CLI-утилиты** из экосистемы .NET (ставятся один раз):

```bash
dotnet tool install --global dotnet-counters
dotnet tool install --global dotnet-gcdump
dotnet tool install --global dotnet-trace
dotnet tool install --global dotnet-dump
```

---

**1. `dotnet-counters` — «приборная панель» (live)**

**Что:** метрики **в реальном времени**, процесс **не останавливается**.

**Когда:** «у weeksync память ползёт вверх», «GC часто мигает», «thread pool забит».

```bash
# список процессов
dotnet-counters ps

# смотреть GC, кучу, аллокации
dotnet-counters monitor --process-id 12345 --counters System.Runtime
```

**Что смотреть:**

| Счётчик | Смысл |
|---|---|
| `gc-heap-size` | сколько заняла куча |
| `gen-0/1/2-gc-count` | как часто собирает поколения |
| `alloc-rate` | скорость аллокаций (байт/сек) |
| `threadpool-queue-length` | очередь в thread pool |

**На собесе:** «первый шаг — **лайв-метрики** без профайлера: `dotnet-counters`».

---

**2. `dotnet-gcdump` — «фото управляемой кучи»**

**Что:** **снимок** managed-объектов в процессе. Легче и быстрее, чем полный дамп. Процесс **не убивается**.

**Когда:** подозрение на **утечку** ([q11](#q11)) — снять **два** снимка с интервалом, сравнить: какой тип вырос и **кто его держит** (retention path / incoming references).

```bash
dotnet-gcdump collect --process-id 12345 -o before.dmp.gcdump
# ... поработать 10–30 мин ...
dotnet-gcdump collect --process-id 12345 -o after.dmp.gcdump
```

Открыть в **Visual Studio** (Memory Usage) или **PerfView** / **dotMemory** — сравнить «до/после».

**На собесе:** «`gcdump` — снимок кучи **без остановки**, для поиска кто удерживает объекты».

---

**3. `dotnet-trace` / PerfView — «видеозапись событий»**

**Что:** запись **потока событий** CLR за период времени: GC-паузы, JIT, аллокации (если включить провайдеры).

**Когда:** «периодические фризы», «долгие GC-паузы», «кто аллоцирует на горячем пути».

```bash
# базовая трассировка GC + runtime
dotnet-trace collect --process-id 12345 --profile gc-collect

# шире — больше провайдеров (файл .nettrace)
dotnet-trace collect --process-id 12345
```

**PerfView** (Windows, бесплатно от Microsoft) — открывает `.nettrace`, показывает **стеки аллокаций**, время в GC. Тяжёлый, но мощный.

**Разница с gcdump:**

| | gcdump | trace |
|---|---|---|
| Время | один момент | интервал |
| Вопрос | «что **лежит** в куче?» | «**что происходило** — GC, alloc, CPU?» |

---

**4. `dotnet-dump` — «морг» процесса (полный дамп)**

**Что:** **весь** процесс на диск (managed + native контекст). Тяжелее gcdump.

**Когда:** прод **упал**, **завис**, нужен разбор **после** инцидента; или gcdump недостаточно.

```bash
dotnet-dump collect --process-id 12345 -o hang.dmp
dotnet-dump analyze hang.dmp    # интерактивный SOS в консоли
```

Внутри `analyze`: команды вроде `dumpheap -stat`, `gcroot <адрес>` — кто держит объект.

**На собесе:** «`dotnet-dump` — **посмертный** анализ, когда процесс уже мёртв или его нельзя трогать профайлером».

---

**5. GUI-профайлеры — то же, но с кнопками**

| Инструмент | Платформа | Зачем |
|---|---|---|
| **Visual Studio** → Diagnostic Tools | Windows, при отладке | память + CPU, удобно локально |
| **dotMemory** (JetBrains) | Win/Mac | красивые графы удержания, сравнение снимков |
| **dotTrace** (JetBrains) | Win/Mac | CPU + timeline, аналог trace с UI |

Если на собесе спросят «чем пользовался» — честно: «локально VS / dotMemory, на сервере — **dotnet-counters + gcdump**». Этого достаточно для Senior-уровня понимания.

---

**Типичный сценарий «память растёт» (связка с q11):**

1. **`dotnet-counters`** — подтвердить рост `gc-heap-size`, не фантазия.
2. **`dotnet-gcdump`** до и после нагрузки — найти тип (`Dictionary`, `byte[]`, `ProjectDto`…).
3. В снимке — **retention path**: статика? кэш? таймер? событие?
4. Если нужны **паузы / alloc rate по времени** — **`dotnet-trace`** или PerfView.
5. Если процесс уже **лежит** — **`dotnet-dump collect`** и разбор offline.

**Что сказать одной фразой на собесе:**

> «Сначала **counters** — живые метрики; для утечек — **два gcdump** и сравнение; для латентности и GC — **trace**; для прод-инцидента — **dotnet-dump**. GUI — VS или dotMemory, если есть лицензия.»

**Подводные камни:**

- Путать **gcdump** (лёгкий снимок кучи) и **dump** (тяжёлый полный дамп).
- Один снимок бесполезен для утечки — нужны **два** с разницей во времени/нагрузке.
- `dotnet-trace` с агрессивными провайдерами сам по себе замедляет процесс — не оставлять на проде надолго.
- `GC.Collect()` перед снимком — искажает картину; для диагностики утечки обычно **не** вызывают.

<a id="q13"></a>
### Вопрос: GC pauses, latency modes, `GCSettings`

**Что такое GC pause**

Пока GC **компактифицирует** кучу или собирает поколение, managed-потоки **останавливаются** — **stop-the-world** (см. [q4](#q4)). Запрос «завис» на 50–200 мс — часто это **пауза Gen 2**, а не сеть.

```
Нормальная работа API:  ████░░██████░░████
                              ↑
                         GC pause — «иголка» на графике латентности
```

| Поколение | Пауза | Когда |
|---|---|---|
| Gen 0 / 1 | обычно **короткая** (мс) | много мелких аллокаций |
| Gen 2 / full GC | **длиннее** | долгоживущие объекты, LOH, давление на память |

**Background GC** ([q3](#q3)) уменьшает паузы Gen 2 — часть работы в фоне, но **полностью** паузы не убирает.

---

**Откуда берутся паузы (практика)**

1. **Много аллокаций** → GC запускается чаще.
2. **Большая куча** → дольше mark + compact.
3. **LOH** ([q2](#q2)) — не компактируется по умолчанию → full GC дороже.
4. **Pinned-объекты** ([q14](#q14)) — мешают compact → фрагментация.

**Главный рычаг в коде:** меньше аллокаций на горячем пути (`Span`, `ArrayPool` — [q9](#q9)), а не «выключить GC».

---

**`GCSettings.LatencyMode` — попросить GC «не мешать»**

Не отключает GC. Говорит: «в критичном участке **отложи** тяжёлую сборку».

```csharp
using System.Runtime;

var prev = GCSettings.LatencyMode;
try
{
    GCSettings.LatencyMode = GCLatencyMode.LowLatency;
    // короткий критичный участок: обработка кадра, realtime-цикл
}
finally
{
    GCSettings.LatencyMode = prev;  // вернуть обратно!
}
```

| Режим | Смысл |
|---|---|
| `Interactive` | дефолт — баланс |
| `LowLatency` | коротко отложить full GC |
| `SustainedLowLatency` | дольше держать низкую латентность (осторожно с памятью) |
| `Batch` | серверный throughput, паузы длиннее, но реже мешают интерактиву |

**Когда реально нужно:** игры, аудио, жёсткий realtime. В обычном ASP.NET **редко** — проще снизить аллокации.

---

**Другие настройки (знать названия)**

| Что | Где | Зачем |
|---|---|---|
| Server vs Workstation GC | `ServerGarbageCollection` в csproj | [q3](#q3) |
| Лимит кучи в контейнере | `DOTNET_GCHeapHardLimit` | не съесть весь pod |
| Принудительный GC | `GC.Collect()` | **только** диагностика / тесты, не в проде |

Диагностика пауз — [q12](#q12): `dotnet-counters` (`gc-heap-size`, `gen-2-gc-count`), `dotnet-trace --profile gc-collect`.

**На собесе:**

> «GC pause — stop-the-world при сборке. Лечится меньшими аллокациями и background GC; `LatencyMode` — временно попросить отложить full GC, не панацея.»

**Подводные камни:**

- `LatencyMode` без `finally` — режим «залипает» на весь процесс.
- `SustainedLowLatency` + утечка/кэш без лимита → память растёт, потом **огромная** пауза.
- Путать «медленный запрос» с cold JIT ([q10](#q10)) или сетью — смотреть trace.
- `GC.Collect()` в проде как «лечение» — маскирует проблему, добавляет паузы.

<a id="q14"></a>
### Вопрос: Pinned objects, `fixed`, `GCHandle`

**Зачем pin (закрепление)**

GC при **компактификации** ([q4](#q4)) **перемещает** объекты в куче и обновляет ссылки. Нативный код (P/Invoke, драйвер, `ReadFile`) держит **сырой указатель** на адрес в памяти — если GC сдвинет объект, указатель сломается.

**Pin** — **термин GC/документации** («закрепить объект»), **не ключевое слово C#**. В коде pin делается так:

| Слово в теории / на собесе | Что пишешь в C# |
|---|---|
| «закрепить объект» (pin) | `fixed (...)` внутри `unsafe` |
| «закрепить объект» (pin) | `GCHandle.Alloc(..., GCHandleType.Pinned)` |
| снять pin | конец блока `fixed` или `handle.Free()` |

`unsafe` сам по себе **не** pin — он только **разрешает** указатели. Pin происходит из‑за **`fixed`** или **`GCHandleType.Pinned`**.

---

**Два способа в C#**

**1. `fixed` — pin без слова `pin` в синтаксисе**

```csharp
byte[] buffer = new byte[1024];
unsafe   // разрешить указатели — НЕ pin
{
    fixed (byte* ptr = buffer)   // ← вот здесь GC pin'ит buffer
    {
        NativeMethods.ProcessBuffer(ptr, buffer.Length);
    }
}  // ← pin снят (buffer снова может двигаться)
```

**2. `GCHandle` — pin **явно** в имени типа**

```csharp
var handle = GCHandle.Alloc(buffer, GCHandleType.Pinned);
try
{
    IntPtr ptr = handle.AddrOfPinnedObject();
    NativeMethods.ProcessBuffer(ptr, buffer.Length);
}
finally
{
    handle.Free();  // обязательно — иначе объект «вечно» pinned
}
```

`GCHandle` с типом `Normal` / `Weak` — **не** pin, а **удержание** объекта от GC (тоже корень — [q5](#q5)).

---

**Почему pin — зло для GC (если злоупотреблять)**

Закреплённый объект **нельзя сдвинуть** → вокруг него «дыры» → **фрагментация** кучи → дорогие аллокации и full GC.

Правило: pin **минимально** и **ненадолго** — внутри `fixed` / `try-finally` с `Free()`.

---

**POH — Pinned Object Heap (.NET 5+)**

Раньше pinned-объекты жили в обычной куче и ломали compact. Сейчас долгоживущие pin часто попадают в **отдельную** POH — меньше вреда основной куче. Для собеса: «знаю, что есть POH; в прикладном коде всё равно pin по минимуму».

---

**Где встречается в жизни**

| Место | Pin |
|---|---|
| Свой P/Invoke | ты явно `fixed` / `GCHandle` |
| `Span<T>` поверх массива | обычно **без** твоего pin — BCL внутри |
| `async` / `await` | state machine может pin для continuations — внутри runtime |
| `stackalloc` | на **стеке**, GC не двигает — pin не нужен |

**Классический кейс — вендорская DLL «как есть»:** криптопровайдер, HSM, PKCS#11, ГОСТ-библиотека, драйвер смарт-карты. Исходников нет, пересобрать под .NET нельзя — только **готовый `.dll`** с C-API: «передай указатель на буфер, верну код ошибки». Ты аллоцируешь `byte[]`, `fixed` / `GCHandle` — pin на время вызова, передаёшь адрес в native, получаешь подпись/шифртекст. `System.Security.Cryptography` в BCL для стандартных алгоритмов уже обёрнут; `fixed` пишешь, когда **сам** цепляешься к чужой native-библиотеке.

В типичном ASP.NET / EF **сам** pin почти не пишешь — это знание «на собес» и для interop/низкоуровневого кода.

**На собесе:**

> «Pin фиксирует адрес объекта для нативного кода. Делаю через `fixed` или `GCHandleType.Pinned`, держу кратко, всегда `Free`. Долгий pin фрагментирует кучу.»

**Подводные камни:**

- Забыли `handle.Free()` — объект остаётся pinned + корень GC.
- Pin **строки** (`fixed (char* p = str)`) — строка в LOH/куче, дорого при частом вызове.
- Держать pin на **большом** массиве минутами — фрагментация.
- Путать pin с «объект не соберётся» — любой живой корень ([q5](#q5), статика, кэш) делает то же без pin.

<a id="q15"></a>
### Вопрос: Метаданные сборок, AssemblyLoadContext, выгрузка

**Сборка (assembly) — не только код**

`.dll` содержит:

| Часть | Что внутри |
|---|---|
| **IL** | скомпилированные методы (то, что JIT/AOT ест) |
| **Метаданные** | имена типов, методов, полей, атрибуты, ссылки на другие сборки |

Рефлексия ([тема 01](../01-csharp-language.md)) читает **метаданные**, не `.cs` файлы. Без метаданных `typeof`, атрибуты, DI по конвенциям не работают.

---

**Зачем `AssemblyLoadContext` (ALC)**

По умолчанию всё грузится в **Default** ALC — одна «песочница» на процесс. Проблемы:

- плагины нельзя **выгрузить** (сборка навсегда в памяти);
- две версии одной библиотеки в одном контексте — конфликт.

**ALC** — отдельный **контекст загрузки**: свои сборки, свои версии, при желании — **выгрузка**.

```
Default ALC          →  основное приложение, ASP.NET
PluginLoadContext    →  plugin.dll (collectible)
AnotherPluginContext →  другая версия Newtonsoft.Json
```

---

**Collectible ALC — выгрузка плагина**

```csharp
var alc = new AssemblyLoadContext("plugins", isCollectible: true);
var asm = alc.LoadFromAssemblyPath(pluginPath);
var plugin = asm.CreateInstance("MyPlugin.EntryPoint");

// ... работа с плагином ...

alc.Unload();                    // запрос на выгрузку
plugin = null;
GC.Collect();
GC.WaitForPendingFinalizers();  // в тестах; в проде — когда нет ссылок, GC сам
```

Выгрузка сработает **только если**:

- ALC создан как **collectible** (`isCollectible: true`);
- **нет ссылок** на типы/объекты из этого ALC извне (ни статика, ни события, ни делегаты в Default ALC).

Если главное приложение подписалось `plugin.Something += Handler` — плагин **не выгрузится** ([q11](#q11)).

---

**Типичные сценарии**

| Сценарий | ALC |
|---|---|
| Обычный ASP.NET | Default, об ALC не думаешь |
| Плагинная система, hot-reload модулей | collectible ALC на плагин |
| Две версии одной DLL в одном процессе | разные ALC |
| Native AOT ([q10](#q10)) | динамическая загрузка **ограничена** |

**`Assembly.Load` vs `LoadFromAssemblyPath`:** первая ищет в текущем ALC / shared runtime; вторая — явный путь (часто для плагинов).

---

**Утечки через ALC (частая боль плагинов)**

```
Plugin ALC  →  объект плагина
       ↑
  событие в Default ALC держит делегат
       ↑
  Default ALC держит Plugin ALC → Unload бесполезен
```

То же, что подписки на события ([q11](#q11)), только масштаб — **вся сборка** не выгружается.

**На собесе:**

> «Сборка = IL + метаданные. ALC изолирует загрузку; collectible ALC позволяет выгрузить плагин, если нет внешних ссылок на его типы. Утечка — событие/статика из хоста держит плагин.»

**Подводные камни:**

- `Unload()` — **запрос**, не мгновенное удаление; нужен сбор мусора и отсутствие ссылок.
- Статическое поле в плагине → живёт, пока жив ALC.
- `AssemblyLoadContext.Default` **не** collectible — основное приложение не выгрузишь.
- Путать с **AppDomain** (.NET Framework) — в .NET Core+ роль частично у ALC.
