## Разбор вопросов

### Вопрос: Пирамида тестирования
**Краткий ответ:** Много быстрых **unit**-тестов в основании, меньше **integration** в середине, мало медленных **e2e** наверху. Антипаттерн — «перевёрнутая пирамида»/«рожок мороженого» (много медленных e2e, мало unit).

---

### Вопрос: Unit test, AAA
**Краткий ответ:** Изолированно проверяет единицу логики. Структура **Arrange-Act-Assert**: подготовка → действие → проверка. Быстрый, детерминированный, без внешних зависимостей.

```csharp
[Fact]
public void Discount_Vip_Returns20Percent()
{
    var calc = new DiscountCalculator();          // Arrange
    var result = calc.For(new Customer { IsVip = true, Orders = 200 }); // Act
    result.Should().Be(0.2m);                      // Assert
}
```

---

### Вопрос: xUnit / NUnit / MSTest
**Краткий ответ:** Все три — фреймворки. xUnit популярен в .NET (новый экземпляр класса на тест → изоляция; `[Fact]`/`[Theory]`). NUnit богаче атрибутами (`[TestCase]`, `[SetUp]`). MSTest — от Microsoft. Принципиально похожи.

---

### Вопрос: Моки/стабы/фейки/спаи
**Краткий ответ (test doubles):**
- **Stub** — возвращает заранее заданные данные (для управления входом).
- **Mock** — проверяет взаимодействия (был ли вызван метод и как).
- **Fake** — рабочая упрощённая реализация (in-memory repository).
- **Spy** — записывает, как его вызывали.
- В .NET: Moq, NSubstitute, FakeItEasy.

---

### Вопрос: Mock vs Stub, state vs interaction testing
**Краткий ответ:**
- **State-based** — проверяем результат/состояние после действия (предпочтительно, устойчивее к рефакторингу).
- **Interaction-based** — проверяем вызовы зависимостей (mock). Нужно для side-effect'ов (отправил email), но хрупко (привязка к реализации).

---

### Вопрос: Что НЕ мокать
**Краткий ответ:** Не мокать типы, которыми не владеете (сторонние библиотеки — оборачивайте своим интерфейсом); не мокать всё подряд (over-mocking делает тесты зеркалом реализации); не мокать простые value-объекты/DTO. Мокают границы (IO, внешние сервисы).

---

### Вопрос: Integration tests (WebApplicationFactory)
**Краткий ответ:** `WebApplicationFactory<Program>` поднимает приложение в памяти (`TestServer`) с реальным DI/pipeline; тестируем через `HttpClient` реальные endpoints, middleware, сериализацию, валидацию. Можно подменять зависимости (БД на тестовую).

---

### Вопрос: Тесты с БД (Testcontainers)
**Краткий ответ:** **Testcontainers** поднимают реальную БД в Docker на время тестов — близко к проду, ловят SQL/миграционные баги. In-memory provider EF быстрый, но **не эквивалентен** реальной БД (отличается поведение, нет проверки SQL) — для интеграционных предпочтительнее Testcontainers/реальная БД.

---

### Вопрос: TDD
**Краткий ответ:** Red-Green-Refactor: написать падающий тест → минимальный код для прохождения → рефакторинг. Помогает дизайну и регрессии. На практике применяют выборочно (там, где помогает).

---

### Вопрос: Тестируемый дизайн
**Краткий ответ:** DI вместо `new`/статики, чистые функции, абстракция времени (`TimeProvider`/`IClock` вместо `DateTime.Now`), абстракция IO. Чем меньше скрытых зависимостей, тем легче тестировать.

---

### Вопрос: Покрытие кода
**Краткий ответ:** Метрика «сколько кода выполнено тестами». Полезна для поиска непокрытых мест, но **не цель сама по себе** (100% покрытие ≠ хорошие тесты). Важнее покрытие критичной логики и качество ассертов.

---

### Вопрос: FluentAssertions
**Краткий ответ:** Читаемые ассерты (`result.Should().Be(...)`, `.Should().Throw<>()`), понятные сообщения об ошибках. Улучшает читаемость и диагностику.

---

### Вопрос: Параметризованные тесты
**Краткий ответ:** `[Theory]` + `[InlineData]`/`[MemberData]`/`[ClassData]` — один тест на множество входов, меньше дублирования.

```csharp
[Theory]
[InlineData(0, false)]
[InlineData(18, true)]
public void IsAdult(int age, bool expected) => Person.IsAdult(age).Should().Be(expected);
```

---

### Вопрос: Async-тесты
**Краткий ответ:** Тестовые методы могут быть `async Task`; await реальной операции, ассерты исключений через `await act.Should().ThrowAsync<>()`. Не использовать `.Result` (дедлок/маскировка).

---

### Вопрос: Snapshot/approval testing
**Краткий ответ:** **Verify** — сравнение результата с сохранённым «снимком» (полезно для сложных объектов/сериализации/API-ответов). Изменения требуют явного подтверждения нового снимка.

---

### Вопрос: Flaky tests
**Краткий ответ:** Нестабильные тесты (то проходят, то нет). Причины: зависимость от времени/порядка/общего состояния, гонки, реальная сеть, `Thread.Sleep`. Лечение: изоляция, детерминизм, абстракция времени, отказ от sleep в пользу ожидания условий.

---

### Вопрос: Нагрузочное тестирование
**Краткий ответ:** см. [тему 11](./11-request-pipeline-http-highload.md). k6, NBomber — проверка поведения под нагрузкой (latency, throughput, деградация), а не корректности.

---

### Вопрос: Mutation testing
**Краткий ответ:** **Stryker.NET** вносит мутации в код и проверяет, ловят ли их тесты. Оценивает **качество** тестов (а не только покрытие): если мутант «выжил» — тесты слабые.
