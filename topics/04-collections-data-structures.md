## Шпаргалка по сложности
| Коллекция | Доступ по индексу | Поиск | Вставка | Удаление |
|-----------|--------|-------|---------|----------|
| `T[]` / `List<T>` | O(1) | O(n) | O(1) аморт. (в конец) | O(n) |
| `Dictionary` | — | O(1) | O(1) | O(1) |
| `HashSet` | — | O(1) | O(1) | O(1) |
| `SortedDictionary` (дерево) | — | O(log n) | O(log n) | O(log n) |
| `SortedList` (массивы) | O(log n) поиск | O(log n) | O(n) | O(n) |
| `LinkedList` | O(n) | O(n) | O(1)* | O(1)* |
| `Queue`/`Stack` | — | O(n) | O(1) аморт. | O(1) |
| `PriorityQueue` | — | — | O(log n) | O(log n) (dequeue) |

\* при наличии узла; поиск узла — O(n).

## Разбор вопросов

### Вопрос: `List<T>` — устройство и рост capacity
**Краткий ответ:**
- Внутри — обычный массив `T[]` + `Count`. Доступ по индексу O(1).
- При добавлении сверх `Capacity` создаётся новый массив **в 2 раза больше**, старый копируется. Поэтому одиночный `Add` — амортизированно O(1), но отдельные вставки дороги (копирование).
- Вставка/удаление в середину — O(n) (сдвиг элементов).

**Подводные камни:** если известен размер — задайте `new List<T>(capacity)`, чтобы избежать многократных перевыделений (и мусора, в т.ч. в LOH при больших массивах).

---

### Вопрос: `Dictionary<K,V>` — хеш-таблица, коллизии
**Краткий ответ:**
- Хеш-таблица: по `GetHashCode()` ключа вычисляется бакет, внутри бакета — цепочка/линейный поиск с `Equals()`. Среднее — O(1).
- Коллизии (разные ключи в один бакет) разрешаются через связанные записи (chaining). Плохой `GetHashCode` (много коллизий) деградирует до O(n).
- Рост: при заполнении увеличивается число бакетов (rehash).

**Подводные камни:**
- Ключ должен быть иммутабельным по полям, влияющим на хеш; изменение ключа после вставки «теряет» элемент.
- Для пользовательских ключей обязательно корректные `Equals`/`GetHashCode` (или используйте `record`).

---

### Вопрос: `HashSet`, `SortedDictionary`, `SortedSet`
**Краткий ответ:**
- **`HashSet<T>`** — множество уникальных, O(1) на операциях, без порядка. Операции множеств (`UnionWith`, `IntersectWith`).
- **`SortedDictionary<K,V>`** — красно-чёрное дерево; ключи отсортированы; O(log n). Быстрые вставки в произвольное место.
- **`SortedSet<T>`** — отсортированное множество (дерево), O(log n), диапазонные запросы (`GetViewBetween`).
- **`SortedList<K,V>`** — два массива (ключи/значения); меньше памяти, быстрый индексный доступ, но вставка O(n). Выгоден при редких вставках и частом переборе по порядку.

---

### Вопрос: `Array` vs `List<T>`, многомерные и jagged
**Краткий ответ:**
- `Array` — фиксированный размер, чуть быстрее и компактнее. `List<T>` — динамический размер поверх массива.
- Многомерный `T[,]` — единый блок памяти. Jagged `T[][]` — массив массивов (строки разной длины), чаще быстрее из-за оптимизаций JIT на одномерных массивах.

---

### Вопрос: `LinkedList<T>` — когда нужен
**Краткий ответ:**
- Двусвязный список: O(1) вставка/удаление **при наличии ссылки на узел**, но O(n) поиск и плохая локальность кэша.
- На практике нужен редко: `List<T>` почти всегда быстрее из-за cache locality. `LinkedList` оправдан при частых вставках/удалениях в середину по уже известному узлу (например, LRU-кэш: dictionary + linked list).

---

### Вопрос: `Queue`, `Stack`, `PriorityQueue`
**Краткий ответ:**
- `Queue<T>` — FIFO (Enqueue/Dequeue), кольцевой буфер.
- `Stack<T>` — LIFO (Push/Pop).
- `PriorityQueue<TElement,TPriority>` (.NET 6+) — бинарная куча; извлечение минимального приоритета O(log n). Не стабильна (равные приоритеты — без гарантии порядка).

---

### Вопрос: Потокобезопасные коллекции
**Краткий ответ:** см. [тему 03](./03-async-multithreading.md). `ConcurrentDictionary`, `ConcurrentQueue/Stack/Bag`, `BlockingCollection` (producer/consumer с блокировкой). Для async-сценариев — `Channel<T>`.

**Подводные камни:** `ConcurrentDictionary.GetOrAdd` может вызвать фабрику несколько раз; перечисление — снапшот.

---

### Вопрос: Иерархия интерфейсов коллекций
**Краткий ответ:**
- `IEnumerable<T>` — только перебор (foreach).
- `ICollection<T>` — + `Count`, `Add`, `Remove`, `Contains`.
- `IList<T>` — + доступ по индексу.
- `IReadOnlyCollection<T>`/`IReadOnlyList<T>` — только чтение (контракт неизменяемости интерфейса, но сама коллекция может меняться через другой интерфейс).

**Совет:** принимайте на вход наиболее общий нужный интерфейс (`IEnumerable<T>`), возвращайте конкретный/read-only по контексту.

---

### Вопрос: Immutable collections
**Краткий ответ:**
- `System.Collections.Immutable`: `ImmutableArray<T>`, `ImmutableList<T>`, `ImmutableDictionary` и др. Любая «модификация» возвращает новую коллекцию (structural sharing для деревьев).
- `ImmutableArray<T>` — обёртка над массивом, быстрый перебор/чтение, но дорогая модификация (полная копия). `ImmutableList<T>` — дерево, O(log n) модификация.
- Потокобезопасны по чтению, удобны для shared-состояния.

---

### Вопрос: `Span<T>` поверх коллекций
**Краткий ответ:** `CollectionsMarshal.AsSpan(list)`, `array.AsSpan()` дают доступ к данным без копирования для высокопроизводительной обработки. `CollectionsMarshal.GetValueRefOrNullRef` — доступ к значению словаря по ссылке без двойного поиска.

---

### Вопрос: Корректные `Equals`/`GetHashCode` для ключей
**Краткий ответ:**
- Контракт: равные объекты обязаны иметь равный хеш; неравные — желательно разные хеши. `GetHashCode` должен быть стабилен, пока объект используется как ключ.
- Используйте `HashCode.Combine(a, b, c)` вместо ручных формул. Или `record` (генерирует корректно).

**Пример:**
```csharp
public override int GetHashCode() => HashCode.Combine(Name, Age);
public override bool Equals(object? o) => o is Person p && p.Name == Name && p.Age == Age;
```

**Подводные камни:** мутабельные поля в хеше → объект «теряется» в словаре после изменения.

---

### Вопрос: `IEnumerator`, `yield`, ленивость
**Краткий ответ:** см. также [тему 01](./01-csharp-language.md). `foreach` использует `IEnumerator<T>` (`MoveNext`/`Current`). `yield return` генерирует ленивый итератор. Перебор не материализует всю коллекцию.

---

### Вопрос: LINQ и материализация, множественный перебор
**Краткий ответ:**
- LINQ-операторы (`Where`, `Select`) ленивы; выполняются при переборе. `ToList()`/`ToArray()`/`Count()`/`First()` материализуют/выполняют.
- **Множественный перебор** одного `IEnumerable` повторяет вычисление (и запрос к БД для `IQueryable`). Если перебираете несколько раз — материализуйте один раз.

**Подводные камни:** `list.Count()` (LINQ) против `list.Count` (свойство) — для `ICollection` LINQ оптимизирован, но привычка вызывать `.Count()` на потенциально ленивом источнике может перебрать всё.
