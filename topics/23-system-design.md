## Базовые концепции (кратко)
- **Масштабирование:** вертикальное (мощнее сервер, есть потолок) vs горизонтальное (больше серверов, нужен stateless + балансировка).
- **Load balancer:** L4 (транспорт) / L7 (HTTP, умная маршрутизация); алгоритмы round-robin/least-connections.
- **Stateless:** состояние выносится в БД/кэш → любой инстанс обслужит запрос → легко масштабировать.
- **CDN:** кэш статики ближе к пользователю.
- **Consistent hashing:** распределение ключей по узлам с минимальной ребалансировкой при добавлении/удалении узла.
- **SPOF:** устранять через репликацию/резервирование.
- **CAP/репликация/шардирование:** см. темы 14, 15.

<a id="q-url-shortener"></a>
### ⭐ Задача: Спроектировать сервис создания коротких ссылок (URL shortener)

**1. Требования**

Функциональные:
- Создать короткую ссылку из длинного URL (опц. кастомный alias, TTL).
- Редирект с короткой на длинную.
- (Опц.) аналитика переходов.

Нефункциональные:
- Низкая латентность редиректа (это критично).
- Высокая доступность (редирект важнее создания).
- Масштаб: сильно **read-heavy** (чтений >> записей, ~100:1).
- Короткие ссылки практически не меняются (хорошо кэшируются).

**2. Оценка масштаба (пример)**
- 100M новых ссылок/месяц ≈ ~40 записей/сек, чтений ~4000/сек (ratio 100:1).
- Хранение: 100M/мес × 5 лет ≈ 6 млрд записей; ~500 байт/запись ≈ ~3 ТБ.
- Длина кода: при base62 (`a-zA-Z0-9`) 7 символов = 62^7 ≈ 3.5 трлн комбинаций — с запасом.

**3. API**
```
POST /shorten        { "url": "...", "alias?": "...", "ttl?": "..." } -> { "short": "https://sho.rt/abc1234" }
GET  /{code}         -> 301/302 Location: <long url>
GET  /{code}/stats   -> аналитика
```

**4. Модель данных**
```
links: code (PK, string) | long_url | created_at | expires_at | owner_id
```
- Доступ по `code` — точечный (key-value подходит идеально). Подойдёт реляционная БД с индексом по PK или key-value (DynamoDB/Redis-as-store) для масштаба.

**5. Генерация кода — ключевое решение**
- **Вариант A — хеш длинного URL** (например, base62 от MD5/SHA, взять первые 7 символов). Минус: коллизии (нужна проверка и разрешение), одинаковый URL → один код (может быть и плюсом).
- **Вариант B — счётчик + base62 кодирование** (auto-increment ID → base62). Без коллизий, но последовательные коды предсказуемы и раскрывают объём; ID-генерация — узкое место в распределёнке.
- **Вариант C — распределённая генерация ID** (Snowflake / выделенные диапазоны ID per-инстанс / Zookeeper). Решает узкое место счётчика, коды не предсказуемо-последовательны.
- **Кастомный alias** — проверка уникальности.

Часто выбирают **B/C**: глобально уникальный ID → base62. Это даёт короткие коды без проверки коллизий.

**6. High-level архитектура**
```
Client → CDN/LB → [Stateless API instances] → Cache (Redis) → DB (key-value/SQL)
                                   ↑
                         ID generator (Snowflake/ranges)
   аналитика: переходы → очередь (Kafka) → асинхронная обработка → хранилище аналитики
```

**7. Горячий путь — редирект (оптимизация)**
- Редирект должен быть очень быстрым → **кэш Redis** `code → long_url` перед БД (cache-aside). Высокий hit-rate, т.к. ссылки иммутабельны.
- Реплики БД для чтения, CDN перед сервисом.
- 301 (permanent, кэшируется браузером, меньше нагрузки) vs 302 (temporary, позволяет считать аналитику на каждом переходе) — trade-off.

**8. Trade-offs и нюансы**
- **301 vs 302:** 301 снижает нагрузку, но мешает точной аналитике и инвалидации.
- **Аналитика** — асинхронно через очередь, чтобы не замедлять редирект.
- **Истечение/чистка** — фоновая джоба или TTL в key-value/Redis.
- **Согласованность:** при создании допустима строгая (нужен уникальный код), при чтении — кэш с eventual обновлением (но коды иммутабельны → проще).
- **Защита:** rate limiting на создание, проверка/блокировка вредоносных URL, приватность кастомных alias.
- **Масштаб записи:** распределённая генерация ID убирает единый счётчик-bottleneck.
