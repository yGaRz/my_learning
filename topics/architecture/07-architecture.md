## Разбор вопросов

<a id="q1"></a>
### Вопрос: Layered (N-tier) архитектура
**Краткий ответ:** Классические слои: Presentation → Application/Business → Data Access → Database. Каждый слой зависит от нижележащего. Просто, понятно, но домен зависит от инфраструктуры (БД), что усложняет тесты и замену.

<a id="q2"></a>
### Вопрос: Clean / Onion / Hexagonal
**Краткий ответ:** Все три — про **инверсию зависимостей к домену**:
- В центре — **Domain** (сущности, правила), не зависит ни от чего.
- Вокруг — **Application** (use cases, интерфейсы портов).
- Снаружи — **Infrastructure** (БД, внешние сервисы) и **Presentation** (API/UI).
- **Hexagonal (Ports & Adapters)**: домен общается с миром через порты (интерфейсы), адаптеры реализуют их (БД, HTTP, очередь).
- **Onion/Clean** — концентрические слои с тем же правилом: зависимости направлены **внутрь**.

**Пример структуры проектов:**
```
Domain            (без зависимостей)
Application        -> Domain (интерфейсы IRepository здесь)
Infrastructure     -> Application, Domain (реализации, EF)
Web/Api            -> Application (+ DI всё собирает)
```

<a id="q3"></a>
### Вопрос: Dependency Rule
**Краткий ответ:** Исходный код зависимостей всегда указывает **внутрь** — к более высокоуровневым политикам (домену). Внутренние слои **не знают** о внешних. Достигается через интерфейсы (порты) в центре и реализации снаружи + DI.

<a id="q4"></a>
### Вопрос: Монолит vs модульный монолит vs микросервисы
**Краткий ответ:**
- **Монолит** — один деплой, простота разработки/отладки/транзакций. Минусы: масштабирование целиком, риск «большого кома грязи».
- **Модульный монолит** — один деплой, но чёткие модули с границами (как bounded contexts) и явными контрактами. Часто **лучший старт**: даёт границы без распределённой сложности.
- **Микросервисы** — независимые деплои/масштабирование/команды, но распределённая сложность (сеть, согласованность, наблюдаемость). Оправданы при масштабе организации/нагрузки.

**Совет на собесе:** «начинать с модульного монолита, выделять микросервисы по необходимости и по границам bounded context».

<a id="q5"></a>
### Вопрос: Anemic vs rich domain model
**Краткий ответ:**
- **Anemic** — сущности только с данными (геттеры/сеттеры), логика в сервисах. Проще, но логика размазана, инварианты не защищены.
- **Rich** — данные + поведение вместе (DDD); инварианты внутри сущностей. Сложнее, но домен выразителен и защищён.
- Выбор зависит от сложности домена: CRUD → анемичная норм; сложная бизнес-логика → rich.

<a id="q6"></a>
### Вопрос: Vertical Slice Architecture
**Краткий ответ:** Организация по фичам (вертикальные срезы), а не по техническим слоям. Каждая фича содержит свой handler, модель, валидацию (часто с MediatR). Минимум общих абстракций, высокая связность по фиче. Хорошо сочетается с CQRS.

<a id="q7"></a>
### Вопрос: DTO vs Entity vs Domain vs ViewModel, маппинг
**Краткий ответ:**
- **Entity (persistence)** — модель хранения (EF).
- **Domain model** — бизнес-объекты с поведением.
- **DTO** — контракт передачи на границе (API), без логики; защищает от over-posting и протечки внутренней модели.
- **ViewModel** — модель под отображение (UI).
- Маппинг между ними: вручную (предсказуемо) или AutoMapper.

**Критика AutoMapper:** скрывает ошибки (рантайм-падения при несоответствии), усложняет отладку и производительность; многие предпочитают ручной/source-generated маппинг (Mapperly).

<a id="q8"></a>
### Вопрос: Cross-cutting concerns
**Краткий ответ:** Сквозная функциональность (логирование, валидация, транзакции, авторизация, кэш, retry) выносится из бизнес-логики:
- middleware/фильтры (ASP.NET),
- MediatR pipeline behaviors,
- декораторы,
- AOP/интерсепторы.

[↑ Наверх](#top)

<a id="q9"></a>
### Вопрос: 12-factor app
**Краткий ответ:** Набор принципов для облачных приложений: конфиг в окружении, stateless-процессы, явные зависимости, логи в stdout, dev/prod parity, порт-биндинг и т.д. Облегчает контейнеризацию и масштабирование.

[↑ Наверх](#top)

<a id="q10"></a>
### Вопрос: Strangler Fig
**Краткий ответ:** Постепенная замена легаси: новый функционал/маршруты перенаправляются на новую систему, старая «удушается» по частям, пока не исчезнет. Снижает риск big-bang переписывания.

<a id="q11"></a>
### Вопрос: ADR (Architecture Decision Records)
**Краткий ответ:** Короткие документы, фиксирующие архитектурное решение, контекст, альтернативы и последствия. Дают историю «почему так» и онбординг.

<a id="q12"></a>
### Вопрос: Conway's Law
**Краткий ответ:** Структура системы повторяет структуру коммуникаций организации. Следствие: границы сервисов/модулей стоит проектировать с учётом структуры команд (и наоборот — «обратный манёвр Конвея»).
