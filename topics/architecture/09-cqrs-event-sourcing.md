## CQRS

<a id="q1"></a>
### Что такое CQRS
**Command Query Responsibility Segregation** — разделение операций **изменения** (Commands) и **чтения** (Queries) на разные модели/пути. Команда меняет состояние и (обычно) не возвращает данные; запрос возвращает данные и не меняет состояние.

<a id="q2"></a>
### CQRS != две базы данных
Это частое заблуждение. Уровни применения:
1. **Уровень кода** — просто разные классы команд/запросов и хендлеров (самый частый и дешёвый вариант).
2. **Уровень модели** — разные модели для записи (rich domain) и чтения (плоские DTO/проекции).
3. **Уровень хранилища** — отдельная read-БД/реплика (только при реальной необходимости масштабирования чтения).

Начинать стоит с уровня 1.

<a id="q3"></a>
### Реализация через MediatR
```csharp
public record CreateOrder(int CustomerId, List<Item> Items) : IRequest<Guid>;

public class CreateOrderHandler(IOrderRepository repo) : IRequestHandler<CreateOrder, Guid>
{
    public async Task<Guid> Handle(CreateOrder cmd, CancellationToken ct)
    {
        var order = Order.Create(cmd.CustomerId, cmd.Items);
        await repo.AddAsync(order, ct);
        return order.Id;
    }
}
```
Сквозная логика (валидация, логирование, транзакции) — через **pipeline behaviors**.

<a id="q4"></a>
### Раздельные read-модели
Под запросы делают денормализованные представления (materialized views, плоские DTO), оптимизированные под конкретные экраны — без джойнов и маппинга домена.

<a id="q5"></a>
### Синхронизация read/write, eventual consistency
Если read-модель отдельная, она обновляется асинхронно (события/проекции) → между записью и появлением в read-модели есть лаг (eventual consistency). Нужно учитывать в UX (например, оптимистичное отображение).

<a id="q6"></a>
### Когда CQRS избыточен
Простой CRUD без расхождения моделей чтения/записи — CQRS добавит церемоний без выгоды. Применяйте при сложной бизнес-логике записи и/или разных требованиях к чтению.

<a id="q7"></a>
### Валидация команд
FluentValidation в pipeline behavior: команда валидируется до хендлера; ошибки → понятный ответ (400/ProblemDetails).

## Event Sourcing

<a id="q8"></a>
### Что такое Event Sourcing
Состояние хранится не как текущий снимок, а как **упорядоченный поток событий** (фактов): `OrderCreated`, `ItemAdded`, `OrderShipped`. Текущее состояние получается **проигрыванием (replay)** событий.

<a id="q9"></a>
### Event Store, replay, snapshots
- **Event Store** — append-only хранилище событий (нельзя менять/удалять, только добавлять).
- **Replay** — восстановление состояния агрегата применением всех его событий по порядку.
- **Snapshots** — периодические снимки состояния, чтобы не проигрывать тысячи событий с нуля (replay от последнего снапшота).

<a id="q10"></a>
### Проекции
Из потока событий строят read-модели (projections) под запросы — фактически CQRS-read часть. Можно иметь много разных проекций одного потока.

<a id="q11"></a>
### Версионирование событий
События живут вечно — схема меняется со временем. Подходы: версии события (`OrderCreatedV2`), upcasting (преобразование старых в новые при чтении), терпимость к лишним/недостающим полям. Нельзя «переименовать» прошлое.

<a id="q12"></a>
### CQRS + ES вместе
ES естественно сочетается с CQRS: write-сторона пишет события, read-сторона строит проекции. Но CQRS можно использовать **без** ES (и чаще так и делают).

<a id="q13"></a>
### Идемпотентность / дедупликация
При доставке событий «as least once» потребители/проекции должны быть идемпотентны (по event id / позиции), чтобы повтор не ломал состояние.

<a id="q14"></a>
### Минусы ES и инструменты
- **Минусы:** высокая сложность, eventual consistency, трудная отладка, версионирование событий, «вечное» хранилище, порог входа команды.
- **Инструменты:** EventStoreDB, Marten (поверх PostgreSQL), Kafka как лог событий, Akka.Persistence.

**Вывод для собеса:** CQRS — часто полезен и дёшев на уровне кода; Event Sourcing — мощный, но дорогой инструмент для конкретных доменов (аудит, финансы, сложные временные процессы), не «по умолчанию».
