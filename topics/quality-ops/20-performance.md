## Разбор вопросов

<a id="q1"></a>
### Вопрос: «Сначала измеряй», преждевременная оптимизация
**Краткий ответ:** «Premature optimization is the root of all evil» (Knuth). Сначала профилируй и найди реальное узкое место (часто это IO/БД/сеть, а не CPU), потом оптимизируй с измерением до/после. Оптимизация без данных — трата времени и риск усложнения.

[↑ Наверх](#top)

---

<a id="q2"></a>
### Вопрос: BenchmarkDotNet
**Краткий ответ:** Стандарт для микробенчмарков в .NET: прогрев, несколько итераций, статистика, учёт JIT/GC, `[MemoryDiagnoser]` (аллокации). Нельзя мерить `Stopwatch`'ом в одном проходе — некорректно.

```csharp
[MemoryDiagnoser]
public class Bench
{
    [Benchmark] public string Concat() => string.Concat("a", "b", "c");
}
```

<a id="q3"></a>
### Вопрос: Аллокации и давление на GC
**Краткий ответ:** см. [тему 02](./02-clr-gc-memory.md). Много аллокаций → частые GC → паузы и снижение throughput. Цель горячего пути — минимум аллокаций (alloc-free), чтобы GC реже запускался.

[↑ Наверх](#top)

<a id="q4"></a>
### Вопрос: Span/Memory/stackalloc/ArrayPool/pooling
**Краткий ответ:** Инструменты снижения аллокаций (см. темы 01, 02, 11): `Span`/`Memory` без копий, `stackalloc` на стеке, `ArrayPool`/`MemoryPool`/`ObjectPool` для переиспользования буферов/объектов, `RecyclableMemoryStream`.

### Вопрос: StringBuilder, string.Create
**Краткий ответ:** см. [тему 01](./01-csharp-language.md). `StringBuilder` при многих конкатенациях; `string.Create` для построения строки в готовый буфер без промежуточных аллокаций.

<a id="q6"></a>
### Вопрос: struct/readonly struct
**Краткий ответ:** Маленькие `readonly struct` избегают аллокаций кучи (живут на стеке/инлайнятся). Но большие структуры дорого копировать (передавать `in`); неосторожное использование может быть медленнее class.

[↑ Наверх](#top)

### Вопрос: Боксинг
**Краткий ответ:** см. [тему 01](./01-csharp-language.md). Скрытый источник аллокаций; избегать через дженерики/Span/не-боксящие API.

[↑ Наверх](#top)

<a id="q8"></a>
### Вопрос: LINQ overhead
**Краткий ответ:** LINQ удобен, но создаёт делегаты/итераторы/замыкания (аллокации) и медленнее ручного цикла. На горячем пути в критичных местах — заменять на циклы/Span. В обычном коде читаемость важнее.

[↑ Наверх](#top)

---

<a id="q9"></a>
### Вопрос: Async overhead, ValueTask
**Краткий ответ:** см. [тему 03](./03-async-multithreading.md). Каждый async-вызов с аллокацией Task на горячем пути даёт оверхед; `ValueTask` и синхронный fast-path снижают аллокации, когда результат часто готов синхронно.

[↑ Наверх](#top)

---

<a id="q10"></a>
### Вопрос: Thread pool starvation
**Краткий ответ:** см. [темы 03, 11](./03-async-multithreading.md). Блокирующий код в async → нехватка потоков → лавинообразный рост latency. Лечение: async all the way, не блокировать.

[↑ Наверх](#top)

---

<a id="q11"></a>
### Вопрос: Bottleneck-анализ
**Краткий ответ:** Определить тип узкого места: **CPU-bound** (профиль CPU), **IO-bound** (ожидание БД/сети — async/кэш/оптимизация запросов), **lock contention** (конкуренция за блокировки), **аллокации/GC**, **сеть/сериализация**. Каждый лечится по-своему.

### Вопрос: N+1 и оптимизация запросов
**Краткий ответ:** см. [тему 13](./13-ef-core-data-access.md). Частая причина «медленного приложения» — неоптимальные запросы к БД (N+1, отсутствие индексов, выборка лишнего), а не C#-код.

<a id="q15"></a>
### Вопрос: Latency vs throughput, перцентили
**Краткий ответ:** **Latency** — время одного запроса; **throughput** — запросов в секунду. Их часто противопоставляют. Измерять не среднее (скрывает хвосты), а **перцентили p50/p95/p99** — они отражают реальный пользовательский опыт и «хвосты».

[↑ Наверх](#top)
