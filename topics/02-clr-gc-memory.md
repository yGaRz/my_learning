## Разбор вопросов

### Вопрос: Стек vs управляемая куча — что где хранится
**Краткий ответ:**
- **Стек** — на поток, фиксированного размера (~1 МБ по умолчанию). Хранит фреймы вызовов: локальные value-переменные, параметры, адреса возврата. Выделение/освобождение — простое сдвигание указателя (быстро), память освобождается автоматически при выходе из метода.
- **Управляемая куча (managed heap)** — общая для процесса. Хранит объекты ссылочных типов и их содержимое. Управляется GC.
- Ссылочная переменная может лежать на стеке, а сам объект — в куче. Value-поля внутри объекта в куче лежат в куче.

**Подводные камни:**
- `StackOverflowException` (например, глубокая рекурсия) нельзя поймать `try/catch` — процесс падает.
- «Структуры всегда на стеке» — миф (см. тему 01).

---

### Вопрос: Поколения GC, LOH, фрагментация
**Краткий ответ:**
- GC построен на гипотезе о поколениях: **большинство объектов живут недолго**.
- **Gen 0** — новые мелкие объекты; собирается чаще всего, дёшево. Пережившие переходят в Gen 1.
- **Gen 1** — буфер между кратко- и долгоживущими.
- **Gen 2** — долгоживущие объекты (синглтоны, кэши, статика); собирается редко, сборка дорогая (full GC).
- **LOH (Large Object Heap)** — объекты ≥ **85 000 байт** (большие массивы, буферы). Собирается вместе с Gen 2 и **по умолчанию не компактируется** → фрагментация. Можно принудительно: `GCSettings.LargeObjectHeapCompactionMode`.

**Подводные камни:**
- Частые большие аллокации (>85 КБ) фрагментируют LOH и вызывают дорогие full GC. Решение — `ArrayPool<T>`/`RecyclableMemoryStream`, переиспользование буферов.
- Мид-лайф кризис: объекты, живущие «средне», доживают до Gen 2 и удорожают сборку.

### Вопрос: Workstation vs Server GC, Concurrent/Background GC
**Краткий ответ:**
- **Workstation GC** — оптимизирован под десктоп/низкую латентность, один heap (исторически). По умолчанию в клиентских приложениях.
- **Server GC** — несколько heap'ов и GC-потоков (по числу ядер), оптимизирован под пропускную способность серверов. По умолчанию в ASP.NET Core на многоядерных машинах. Больше памяти, выше throughput.
- **Background/Concurrent GC** — сборка Gen 2 выполняется в фоновом потоке, минимизируя паузы (stop-the-world).

**Настройка:** `<ServerGarbageCollection>true</ServerGarbageCollection>`, `<ConcurrentGarbageCollection>` в csproj или через env `DOTNET_gcServer`.

**Подводные камни:**
- В контейнерах с лимитами памяти Server GC может потреблять много RAM (heap на ядро). Учитывайте лимиты (`DOTNET_GCHeapHardLimit`, cgroup awareness).

### Вопрос: Как работает mark-and-sweep + compaction
**Краткий ответ:**
1. **Mark** — обход графа объектов от корней (roots); достижимые помечаются как «живые».
2. **Sweep** — недостижимые считаются мусором, их память освобождается.
3. **Compact** — живые объекты сдвигаются вместе, устраняя «дыры»; обновляются ссылки. Это устраняет фрагментацию и делает аллокацию дешёвой (bump pointer). LOH по умолчанию не компактируется.

**Подводные камни:**
- Во время компактификации ссылки меняются — поэтому «pinned» объекты (см. ниже) мешают и фрагментируют кучу.

---

### Вопрос: Корни GC (roots)
**Краткий ответ:** Корни — точки входа, от которых GC считает объекты достижимыми:
- локальные переменные в активных фреймах стека и регистрах CPU;
- статические поля;
- объекты в очереди финализации;
- `GCHandle` (в т.ч. pinned).

Объект жив, пока достижим из любого корня.

**Подводные камни:**
- Статическое поле, ссылающееся на большой объект/коллекцию, держит его весь срок жизни приложения — классическая «утечка».

---

### Вопрос: Финализация, очередь финализации, `GC.SuppressFinalize`
**Краткий ответ:**
- Объект с финализатором (`~Class`) при создании регистрируется в finalization queue. Когда становится недостижим, попадает в f-reachable queue, и отдельный поток-финализатор вызывает финализатор. Только после этого объект собирается — то есть он **переживает минимум один лишний GC**.
- `GC.SuppressFinalize(this)` в `Dispose()` снимает объект с очереди финализации (раз ресурсы уже освобождены детерминированно) — это удешевляет сборку.

**Подводные камни:**
- Финализатор недетерминирован (неизвестно когда сработает), не гарантирован порядок, не должен трогать другие управляемые объекты (они могут быть уже собраны).
- Исключение в финализаторе роняет процесс.

---

### Вопрос: `IDisposable` и Dispose pattern vs финализатор
**Краткий ответ:** см. подробный Dispose pattern в [теме 01](./01-csharp-language.md). Кратко:
- `Dispose()` — детерминированное освобождение (вызывает разработчик/`using`).
- Финализатор — недетерминированная страховка только для прямого владения неуправляемым ресурсом.
- Идеально: реализовать `IDisposable`, а финализатор добавлять только если есть неуправляемый хэндл; в `Dispose` вызывать `GC.SuppressFinalize`.

**Подводные камни:** двойной вызов `Dispose` должен быть безопасен (флаг `_disposed`).

### Вопрос: Boxing/unboxing — где аллокации, как избежать
**Краткий ответ:** см. [тему 01](./01-csharp-language.md#). Боксинг = аллокация в куче при упаковке value-типа в `object`/интерфейс. Избегать: дженерики, `Span<T>`, не-боксящие API, переопределённые `Equals`/`GetHashCode` у структур, кэширование забоксенных значений.

**Пример скрытого боксинга:**
```csharp
int x = 5;
object o = x;                     // boxing
IComparable c = x;                // boxing (интерфейс на value-типе)
Console.WriteLine($"{x}");        // в старых версиях боксинг аргумента; в новых — оптимизировано
```

### Вопрос: `Span<T>`, `Memory<T>`, `stackalloc`, `ArrayPool<T>`
**Краткий ответ:**
- **`Span<T>`/`ReadOnlySpan<T>`** — «окно» в непрерывную область памяти (массив, стек, нативная память) без аллокации и копирования. `ref struct` — только стек, нельзя в полях/async (см. тему 01).
- **`Memory<T>`** — аналог Span, который можно хранить в полях и использовать в `async` (через `.Span` получают Span синхронно).
- **`stackalloc`** — выделение буфера на стеке (`Span<byte> buf = stackalloc byte[256];`) без нагрузки на GC; только для небольших размеров.
- **`ArrayPool<T>`** — пул переиспользуемых массивов; снижает аллокации/LOH для буферов. Обязательно `Return`.

**Пример:**
```csharp
byte[] buffer = ArrayPool<byte>.Shared.Rent(4096);
try
{
    int read = stream.Read(buffer, 0, buffer.Length);
    Process(buffer.AsSpan(0, read));
}
finally
{
    ArrayPool<byte>.Shared.Return(buffer);
}
```

**Подводные камни:** после `Return` нельзя пользоваться массивом; `stackalloc` большого размера → переполнение стека.

### Вопрос: JIT, AOT, Tiered Compilation, ReadyToRun
**Краткий ответ:**
- **JIT** — IL компилируется в нативный код при первом вызове метода.
- **Tiered Compilation** — сначала быстрый незаоптимизированный код (Tier 0, быстрый старт), затем для «горячих» методов перекомпиляция в оптимизированный (Tier 1). **Tiered PGO** добавляет профиль-ориентированную оптимизацию.
- **ReadyToRun (R2R)** — AOT-предкомпиляция при публикации: быстрее старт за счёт большего размера; JIT всё ещё доступен.
- **Native AOT** — полная компиляция в нативный бинарник без JIT/рантайм-JIT: быстрый старт, малое потребление памяти, но без рефлексии-зависимого кода и динамической загрузки сборок.

**Подводные камни:** Native AOT/trimming ломают рефлексию и динамику — нужны source generators (см. тему 01).

### Вопрос: Утечки памяти в .NET — причины
**Краткий ответ:** GC не спасает от логических утечек — объект «живёт», пока достижим:
- **Статические коллекции/поля**, в которые добавляют и не удаляют.
- **Незакрытые подписки на события**: паблишер держит подписчика через делегат (`+=` без `-=`).
- **Замыкания**, захватывающие `this` и попавшие в долгоживущие структуры/таймеры.
- **`IDisposable`, который не диспоузят** (нативные хэндлы, соединения).
- **Кэши без вытеснения** (растут бесконечно).
- **Таймеры/`Timer`**, удерживающие колбэк.

**Диагностика:** снять два дампа памяти (`dotnet-gcdump`/dotMemory), сравнить рост объектов, найти кто их держит (retention path).

### Вопрос: Инструменты диагностики
**Краткий ответ:**
- **dotnet-counters** — лайв-метрики (GC, heap, alloc rate, thread pool).
- **dotnet-gcdump** — снимок управляемой кучи без остановки.
- **dotnet-trace / PerfView** — трассировка событий, аллокации, паузы GC.
- **dotMemory / dotTrace (JetBrains)**, **Visual Studio Diagnostic Tools** — профайлеры памяти и CPU.
- **dotnet-dump** — полный дамп для посмертного анализа.

### Вопрос: GC pauses, latency modes, `GCSettings`
**Краткий ответ:**
- Сборка (особенно Gen 2 в не-background режиме) приостанавливает managed-потоки (stop-the-world) → латентные «иголки».
- `GCSettings.LatencyMode` (`LowLatency`, `SustainedLowLatency`) — попросить GC реже делать полные сборки в критичных участках.
- На горячем пути цель — минимизировать аллокации, чтобы GC просто не запускался часто.

**Подводные камни:** LatencyMode не отключает GC, лишь меняет поведение; злоупотребление ведёт к росту памяти.

---

### Вопрос: Pinned objects, `fixed`, `GCHandle`
**Краткий ответ:**
- При interop с нативным кодом объект нужно «закрепить» (pin), чтобы GC не переместил его при компактификации. `fixed` (в unsafe-контексте) или `GCHandle.Alloc(obj, GCHandleType.Pinned)`.
- Закреплённые объекты мешают компактификации → фрагментация. Закреплять ненадолго.
- **POH (Pinned Object Heap)** (.NET 5+) — отдельная куча для закреплённых объектов, уменьшает фрагментацию.

---

### Вопрос: Метаданные сборок, AssemblyLoadContext, выгрузка
**Краткий ответ:**
- Сборка (assembly) содержит IL + метаданные (типы, члены, атрибуты).
- **AssemblyLoadContext (ALC)** — изолированный контекст загрузки сборок; позволяет загружать/выгружать плагины (collectible ALC) и иметь разные версии одной библиотеки.
- Выгрузка возможна только для collectible ALC, когда нет живых ссылок на его типы.

**Подводные камни:** утечки через ALC — частая проблема плагинных систем (ссылки/события из основного контекста удерживают плагин, мешая выгрузке).
