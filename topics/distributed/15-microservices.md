## Разбор вопросов

<a id="q1"></a>
### Вопрос: Микросервисы vs монолит, когда НЕ дробить
**Краткий ответ:** см. [тему 07](./07-architecture.md). Микросервисы дают независимость деплоя/масштабирования/команд ценой распределённой сложности. НЕ дробить: маленькая команда/продукт, неясные границы домена, отсутствие зрелой инфраструктуры (CI/CD, observability). Старт — модульный монолит.

<a id="q2"></a>
### Вопрос: Границы сервисов
**Краткий ответ:** По **bounded context** из DDD (см. тему 08). Сервис владеет своими данными, имеет высокую внутреннюю связность и слабую связь с другими. Плохие границы → «распределённый монолит» (худшее из двух миров).

[↑ Наверх](#top)

<a id="q3"></a>
### Вопрос: Sync vs async взаимодействие
**Краткий ответ:**
- **Синхронное** (HTTP/gRPC) — просто, но создаёт временную связанность и каскадные сбои (один медленный сервис тормозит цепочку).
- **Асинхронное** (события через брокер) — развязка, устойчивость, но eventual consistency и сложнее отладка.
- Часто комбинируют: команды/запросы синхронно, интеграция/нотификации — через события.

[↑ Наверх](#top)

---

<a id="q4"></a>
### Вопрос: CAP, PACELC
**Краткий ответ:** см. [тему 14](./14-databases-sql.md). При сетевом разделении выбор C vs A. PACELC: иначе (Else) — latency vs consistency. В распределённых системах обычно осознанно принимают eventual consistency ради доступности.

[↑ Наверх](#top)

<a id="q5"></a>
### Вопрос: Strong vs eventual consistency
**Краткий ответ:** Strong — все видят актуальные данные сразу (дорого в распределёнке). Eventual — со временем согласуется; между сервисами/агрегатами почти всегда eventual (через события). Нужно проектировать UX и бизнес-процессы с учётом лага.

[↑ Наверх](#top)

---

<a id="q6"></a>
### Вопрос: Почему 2PC (two-phase commit) плохо
**Краткий ответ:** Распределённая транзакция через координатора (prepare/commit) блокирует ресурсы, плохо масштабируется, координатор — single point of failure, при сбоях возможны зависшие транзакции. В микросервисах предпочитают Saga + eventual consistency.

[↑ Наверх](#top)

---

<a id="q7"></a>
### Вопрос: Saga
**Краткий ответ:** Бизнес-транзакция как последовательность локальных транзакций в разных сервисах, связанных событиями; при сбое выполняются **компенсирующие** действия (отмена ранее сделанного).
- **Хореография** — сервисы реагируют на события друг друга (без центра); проще, но логика размазана.
- **Оркестрация** — центральный оркестратор управляет шагами; нагляднее, но добавляет компонент.

```
CreateOrder -> ReserveStock -> ProcessPayment
   при сбое оплаты: CompensateStock (вернуть резерв) + CancelOrder
```

<a id="q8"></a>
### Вопрос: Outbox pattern
**Краткий ответ:** Проблема: нельзя атомарно записать в БД и опубликовать событие в брокер (двойная запись). Решение: в **той же транзакции** с бизнес-данными пишем событие в таблицу `Outbox`; отдельный процесс (relay) читает outbox и публикует в брокер с гарантией at-least-once. Гарантирует, что событие не потеряется.

[↑ Наверх](#top)

---

<a id="q9"></a>
### Вопрос: Inbox / идемпотентность потребителя
**Краткий ответ:** При at-least-once доставке потребитель может получить дубликаты. **Inbox**: хранить обработанные message id и пропускать повторы (дедупликация). Обработчики должны быть идемпотентны (повтор не меняет результат).

[↑ Наверх](#top)

<a id="q10"></a>
### Вопрос: Resilience (retry, circuit breaker, bulkhead, timeout)
**Краткий ответ:**
- **Retry** с экспоненциальной задержкой и jitter — для транзиентных сбоев (только идемпотентные операции!).
- **Circuit breaker** — при череде ошибок «размыкает» цепь, быстро отклоняет запросы, давая сервису восстановиться.
- **Timeout** — не ждать вечно.
- **Bulkhead** — изоляция пулов ресурсов, чтобы сбой одной зависимости не утопил всё.
- **Fallback** — деградация с запасным ответом.
- В .NET — `Microsoft.Extensions.Http.Resilience`/Polly.

[↑ Наверх](#top)

---

<a id="q11"></a>
### Вопрос: Идемпотентность
**Краткий ответ:** Операция, дающая тот же результат при повторе. Ключи идемпотентности для POST, проверка состояния перед действием. Критично для retry и at-least-once доставки.

[↑ Наверх](#top)

---

<a id="q12"></a>
### Вопрос: API Gateway, BFF
**Краткий ответ:**
- **API Gateway** — единая точка входа: маршрутизация, агрегация, auth, rate limiting, TLS (YARP, Ocelot, облачные).
- **BFF (Backend for Frontend)** — отдельный gateway под нужды конкретного клиента (web/mobile), агрегирует и адаптирует данные.

**Краткий ответ:** Динамический поиск адресов сервисов (Consul, Eureka, k8s DNS/Service). Нужен, т.к. экземпляры эфемерны (масштабирование, перезапуски).

**Краткий ответ:** Обратная совместимость контрактов, параллельная работа версий, экспанд-контракт миграции схемы. Не ломать существующих потребителей (см. тему 12).

<a id="q16"></a>
### Вопрос: Data ownership, database-per-service
**Краткий ответ:** Каждый сервис владеет своими данными и своей БД; другие обращаются только через API/события, не лезут в чужую БД напрямую. Иначе — скрытая связанность.

[↑ Наверх](#top)

<a id="q17"></a>
### Вопрос: Service mesh
**Краткий ответ:** Инфраструктурный слой (Istio/Linkerd) через sidecar-прокси: mTLS, retry/timeout, балансировка, телеметрия — вне кода приложения. Полезен при большом числе сервисов; добавляет сложность эксплуатации.
