## Разбор вопросов

<a id="q1"></a>
### Вопрос: Зачем брокер сообщений
**Краткий ответ:** Развязка отправителя и получателя (во времени и пространстве), буферизация пиков нагрузки, масштабирование потребителей, надёжная асинхронная интеграция, устойчивость (получатель может быть временно недоступен).

[↑ Наверх](#top)

---

<a id="q2"></a>
### Вопрос: Queue vs publish/subscribe
**Краткий ответ:**
- **Queue** — сообщение обрабатывает **один** потребитель (work distribution, конкурирующие потребители).
- **Pub/Sub (topic)** — сообщение получают **все** подписчики (broadcast событий).

[↑ Наверх](#top)

---

<a id="q3"></a>
### Вопрос: RabbitMQ
**Краткий ответ:** Брокер с моделью AMQP. Producer публикует в **exchange**, тот по правилам маршрутизирует в **очереди**, откуда читают потребители.
- **Типы exchange:** direct (по точному routing key), topic (по шаблону), fanout (всем), headers.
- Умная маршрутизация, гибкость, push-модель. Сообщение удаляется после ack.

[↑ Наверх](#top)

---

<a id="q4"></a>
### Вопрос: Kafka
**Краткий ответ:** Распределённый **лог** событий. Топик делится на **партиции**; сообщения упорядочены внутри партиции и хранятся (retention), читаются по **offset**.
- **Consumer group**: партиции распределяются между потребителями группы (параллелизм). Один партишн — одному потребителю в группе.
- Pull-модель, перечитывание (replay), очень высокая пропускная способность, log compaction.

[↑ Наверх](#top)

---

<a id="q5"></a>
### Вопрос: RabbitMQ vs Kafka
**Краткий ответ:**
- **RabbitMQ** — гибкая маршрутизация, очереди задач, низкая латентность, сообщения «съедаются». Хорош для команд/RPC/распределения работы.
- **Kafka** — высокий throughput, хранение и переигрывание событий, потоковая обработка, event sourcing/аналитика. Хорош для event streaming и большого объёма.

[↑ Наверх](#top)

---

<a id="q6"></a>
### Вопрос: Гарантии доставки
**Краткий ответ:**
- **At-most-once** — возможна потеря, без дубликатов.
- **At-least-once** — без потерь, возможны дубликаты (самый частый выбор) → нужна идемпотентность.
- **Exactly-once** — сложно/дорого; в Kafka достижимо для определённых сценариев (транзакции, idempotent producer), но в общем случае правильнее «at-least-once + идемпотентный потребитель».

[↑ Наверх](#top)

### Вопрос: Идемпотентность/дедупликация
**Краткий ответ:** см. [тему 15](./15-microservices.md) (inbox). Хранить обработанные message id, проверять перед обработкой; делать обработчики идемпотентными.

<a id="q8"></a>
### Вопрос: Порядок сообщений
**Краткий ответ:** Глобальный порядок дорог. Kafka гарантирует порядок **внутри партиции** → класть связанные сообщения (по ключу, напр. по orderId) в одну партицию. RabbitMQ — порядок в одной очереди, но конкурирующие потребители и retry его нарушают.

[↑ Наверх](#top)

---

<a id="q9"></a>
### Вопрос: Ack/nack, prefetch
**Краткий ответ:** Потребитель подтверждает (ack) после успешной обработки → брокер удаляет/сдвигает offset. nack/reject → повтор или DLQ. **Prefetch** ограничивает число неподтверждённых сообщений у потребителя (баланс throughput/память/справедливость).

[↑ Наверх](#top)

---

<a id="q10"></a>
### Вопрос: DLQ, retry с backoff
**Краткий ответ:** **Dead Letter Queue** — куда отправляются сообщения, которые не удалось обработать (после N попыток/TTL/reject). Retry с экспоненциальным backoff (часто через retry-очереди с TTL) перед уходом в DLQ. DLQ мониторят и разбирают вручную/автоматически.

[↑ Наверх](#top)

---

<a id="q11"></a>
### Вопрос: Poison messages
**Краткий ответ:** Сообщение, которое всегда падает при обработке и бесконечно ретраится, блокируя очередь. Лечение: лимит попыток → DLQ, чтобы не зацикливаться.

[↑ Наверх](#top)

---

<a id="q12"></a>
### Вопрос: Outbox
**Краткий ответ:** см. [тему 15](./15-microservices.md). Атомарная запись бизнес-данных и события в одну БД-транзакцию, затем надёжная публикация relay-процессом (решает проблему двойной записи).

[↑ Наверх](#top)

---

<a id="q13"></a>
### Вопрос: Backpressure
**Краткий ответ:** Если потребитель не успевает, нужно притормозить производителя/ограничить очередь (bounded), иначе переполнение памяти/брокера. Prefetch, лимиты очередей, мониторинг лага потребителя (consumer lag в Kafka).

[↑ Наверх](#top)

---

<a id="q14"></a>
### Вопрос: Сериализация, Schema Registry
**Краткий ответ:** Сообщения сериализуют (JSON/Protobuf/Avro). **Schema Registry** (для Avro/Protobuf) централизует схемы и контролирует совместимость версий — безопасная эволюция контрактов сообщений.

[↑ Наверх](#top)

<a id="q15"></a>
### Вопрос: MassTransit / NServiceBus
**Краткий ответ:** Абстракции поверх брокеров (RabbitMQ/Azure SB/Kafka): саги, retry/redelivery, DLQ, outbox, сериализация, тестирование — меньше boilerplate, единый API.

[↑ Наверх](#top)

---

<a id="q16"></a>
### Вопрос: Event-driven, события vs команды vs сообщения
**Краткий ответ:**
- **Команда** — указание сделать что-то (один получатель, может отклонить).
- **Событие** — факт, что что-то произошло (много подписчиков, в прошедшем времени).
- **Сообщение** — общий транспортный конверт. Event-driven architecture строится на публикации/подписке событий.
