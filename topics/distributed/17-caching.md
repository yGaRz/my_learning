## Разбор вопросов

<a id="q1"></a>
### Вопрос: Зачем кэш, что кэшировать
**Краткий ответ:** Снизить латентность и нагрузку на БД/сервисы за счёт хранения часто читаемых, редко меняющихся, дорогих в вычислении данных. Не кэшировать: часто меняющееся, персональное/чувствительное без нужды, дешёвое в получении.

[↑ Наверх](#top)

---

<a id="q2"></a>
### Вопрос: In-memory vs distributed (Redis)
**Краткий ответ:**
- **In-memory (`IMemoryCache`)** — в памяти процесса: быстрее всего, но не разделяется между инстансами и теряется при рестарте; риск рассинхрона при нескольких репликах.
- **Distributed (`IDistributedCache`/Redis)** — общий для всех инстансов, переживает рестарт приложения, но сетевой хоп + сериализация.

[↑ Наверх](#top)

---

<a id="q3"></a>
### Вопрос: Уровни кэширования
**Краткий ответ:** Клиент (браузер) → CDN → reverse proxy → приложение (in-memory/distributed) → БД (буферный кэш/план). Чем ближе к клиенту, тем дешевле, но сложнее инвалидация.

[↑ Наверх](#top)

---

<a id="q4"></a>
### Вопрос: HTTP-кэширование
**Краткий ответ:** `Cache-Control` (max-age, no-store, private/public), `ETag` + `If-None-Match` → 304 Not Modified (без передачи тела). В ASP.NET Core: **Output Caching** (.NET 7+, кэш на сервере), Response Caching. Снимает нагрузку с приложения (см. тему 11).

[↑ Наверх](#top)

<a id="q5"></a>
### Вопрос: Стратегии кэширования
**Краткий ответ:**
- **Cache-Aside (lazy)** — приложение само: смотрит кэш, при промахе читает БД и кладёт в кэш. Самая частая.
- **Read-Through** — кэш сам подгружает из источника при промахе.
- **Write-Through** — запись идёт в кэш и синхронно в БД (консистентно, медленнее запись).
- **Write-Behind (write-back)** — запись в кэш, в БД асинхронно (быстро, риск потери).

```csharp
var user = await cache.GetOrCreateAsync(key, async e =>
{
    e.AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5);
    return await db.Users.FindAsync(id);
});
```

<a id="q6"></a>
### Вопрос: Инвалидация
**Краткий ответ:** «Одна из двух сложных задач в CS». Подходы: TTL (по времени), явная инвалидация при изменении (удаление/обновление ключа), версионирование ключей, событийная инвалидация (pub/sub при изменении данных). Баланс между свежестью и сложностью.

[↑ Наверх](#top)

<a id="q7"></a>
### Вопрос: TTL, sliding vs absolute
**Краткий ответ:**
- **Absolute expiration** — истекает в фиксированный момент.
- **Sliding expiration** — продлевается при каждом обращении (но может «вечно» жить у активных ключей — комбинируют с абсолютным максимумом).

[↑ Наверх](#top)

<a id="q8"></a>
### Вопрос: Eviction policies
**Краткий ответ:** При нехватке памяти вытеснение по политике: **LRU** (давно не использованные), **LFU** (редко используемые), random, TTL-based. У `MemoryCache` — ограничение по size; у Redis — `maxmemory-policy`. Обязательно ограничивать размер, иначе утечка памяти.

[↑ Наверх](#top)

---

<a id="q9"></a>
### Вопрос: Cache stampede (thundering herd)
**Краткий ответ:** При истечении популярного ключа множество запросов одновременно идут в БД пересчитывать значение → пик нагрузки. Решения: блокировка/`SemaphoreSlim` на пересчёт (только один считает), early/probabilistic refresh (обновлять до истечения), `HybridCache` (встроенная защита от stampede).

[↑ Наверх](#top)

<a id="q10"></a>
### Вопрос: Stale data, согласованность
**Краткий ответ:** Кэш по природе может содержать устаревшие данные. Нужно решить допустимый уровень устаревания (TTL), стратегию инвалидации, и где строгая консистентность критична — не кэшировать или write-through.

<a id="q11"></a>
### Вопрос: Redis — структуры, persistence, кластеризация
**Краткий ответ:**
- Структуры: strings, hashes, lists, sets, sorted sets, streams, HyperLogLog, bitmaps.
- **Persistence:** RDB (снапшоты) и AOF (журнал команд) — баланс durability/производительности.
- **Кластеризация:** sharding по слотам, репликация, Sentinel/Cluster для HA.
- Однопоточная модель команд (атомарность операций), pub/sub, Lua-скрипты.

[↑ Наверх](#top)

<a id="q12"></a>
### Вопрос: Распределённые блокировки (Redlock)
**Краткий ответ:** Redis можно использовать для распределённых локов (SET NX с TTL / алгоритм Redlock). Но это **не строгая** гарантия (споры о корректности при сетевых задержках/GC-паузах) — для критичных к корректности сценариев нужны более надёжные механизмы (fencing tokens, consensus).

[↑ Наверх](#top)

<a id="q13"></a>
### Вопрос: Сериализация, размер значений
**Краткий ответ:** В distributed cache значения сериализуются (JSON/binary/protobuf) — учитывайте стоимость и размер. Большие значения дороги по сети и памяти; разбивайте/сжимайте при необходимости.

[↑ Наверх](#top)

---

<a id="q14"></a>
### Вопрос: Hot keys
**Краткий ответ:** Один «горячий» ключ концентрирует нагрузку на один узел/партицию. Лечение: локальная реплика in-memory поверх distributed, репликация ключа, шардирование значения.

[↑ Наверх](#top)

---

<a id="q15"></a>
### Вопрос: HybridCache (.NET 9)
**Краткий ответ:** Новый API, объединяющий L1 (in-memory) + L2 (distributed) кэш с защитой от stampede (single-flight), тегами для инвалидации и унифицированным API. Упрощает многоуровневое кэширование.
