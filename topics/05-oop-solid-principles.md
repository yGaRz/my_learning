## Разбор вопросов

### Вопрос: 4 столпа ООП
**Краткий ответ:**
- **Инкапсуляция** — сокрытие внутреннего состояния, доступ через контролируемый интерфейс; защита инвариантов.
- **Наследование** — переиспользование/специализация через иерархию типов (is-a).
- **Полиморфизм** — единый интерфейс для разных реализаций (виртуальные методы, интерфейсы).
- **Абстракция** — выделение существенного, скрытие деталей (интерфейсы/абстрактные классы).

**Подводные камни:** наследование ради переиспользования кода (а не is-a отношения) — частая ошибка; ведёт к хрупким иерархиям.

---

### Вопрос: Наследование vs композиция
**Краткий ответ:**
- «**Favor composition over inheritance**»: композиция (has-a, делегирование) гибче, не создаёт жёсткой связи с базовым классом, легче тестировать и менять поведение в рантайме.
- Наследование уместно при истинном is-a и стабильной иерархии. Иначе — интерфейсы + композиция (например, стратегия вместо подклассов).

**Пример:**
```csharp
// Вместо иерархии подклассов с разным поведением:
class Report(IExportStrategy export) { public void Save() => export.Export(this); }
```

---

### Вопрос: S — Single Responsibility
**Краткий ответ:** У класса должна быть **одна причина для изменения** — одна ответственность/один «актор», ради которого он меняется. `OrderService`, который и считает цену, и шлёт email, и пишет в БД — нарушает SRP.

**Подводные камни:** не доводить до абсурда (класс на каждый метод); ответственность — на уровне причины изменения, а не строки кода.

---

### Вопрос: O — Open/Closed
**Краткий ответ:** Открыт для расширения, закрыт для модификации. Новое поведение добавляется новым кодом (новая реализация интерфейса/стратегия), а не правкой существующего и протестированного.

**Пример:** добавление нового способа оплаты = новый класс `IPaymentMethod`, без правки калькулятора.

---

### Вопрос: L — Liskov Substitution
**Краткий ответ:** Объект подтипа должен подставляться вместо базового без нарушения корректности. Подтип не должен усиливать предусловия или ослаблять постусловия, бросать неожиданные исключения.

**Классика:** `Square : Rectangle` — установка ширины меняет высоту, ломая ожидания кода, работающего с `Rectangle`. Нарушение LSP.

**Подводные камни:** методы, бросающие `NotSupportedException` (например `ReadOnlyList.Add`), формально нарушают LSP.

---

### Вопрос: I — Interface Segregation
**Краткий ответ:** Лучше много специализированных интерфейсов, чем один «толстый». Клиент не должен зависеть от методов, которые не использует.

**Пример:** вместо `IRepository` с 20 методами — `IReadRepository`/`IWriteRepository` под нужды клиентов.

---

### Вопрос: D — Dependency Inversion vs DI vs IoC
**Краткий ответ:**
- **DIP (принцип)**: модули верхнего уровня зависят от **абстракций**, а не от конкретных реализаций; абстракции не зависят от деталей.
- **IoC (инверсия управления)** — общий принцип передачи контроля над созданием/потоком наружу (контейнеру/фреймворку).
- **DI (внедрение зависимостей)** — конкретная техника реализации IoC: зависимости передаются извне (обычно через конструктор).

**Пример:**
```csharp
class OrderService(IEmailSender sender) { ... } // зависит от абстракции, а не от SmtpSender
```

---

### Вопрос: DRY, KISS, YAGNI
**Краткий ответ:**
- **DRY** — не дублируй знание (но не путать случайное совпадение кода с одним знанием — преждевременное обобщение тоже вредно).
- **KISS** — проще = лучше; избегай ненужной сложности.
- **YAGNI** — не реализуй «на будущее» то, что не нужно сейчас.

**Подводные камни:** агрессивный DRY ведёт к преждевременным абстракциям и связности; иногда «дублирование дешевле неправильной абстракции».

---

### Вопрос: Закон Деметры (Law of Demeter)
**Краткий ответ:** «Не разговаривай с незнакомцами»: метод должен вызывать методы только своих полей/параметров/созданных объектов, а не лезть вглубь чужих объектов (`a.GetB().GetC().Do()` — train wreck). Снижает связность.

---

### Вопрос: Tell, Don't Ask
**Краткий ответ:** Не вытаскивай данные из объекта, чтобы принять за него решение — поручи объекту сделать это (поведение рядом с данными). Помогает избегать анемичных моделей.

---

### Вопрос: Coupling и Cohesion
**Краткий ответ:**
- **Coupling (связанность)** между модулями — стремимся к низкой (loose coupling).
- **Cohesion (сцепление/связность внутри)** модуля — стремимся к высокой (элементы модуля логически связаны одной задачей).
- Цель дизайна: **low coupling, high cohesion**.

---

### Вопрос: Абстрактные классы vs интерфейсы
**Краткий ответ:**
- **Интерфейс** — контракт без состояния (с C# 8 — default-методы, но без полей). Класс реализует много интерфейсов.
- **Абстрактный класс** — частичная реализация + общее состояние, единичное наследование.
- Выбор: «can-do/роль» → интерфейс; «is-a + общий код/состояние» → абстрактный класс.

---

### Вопрос: Виртуальные методы, `sealed`, `new`
**Краткий ответ:**
- `virtual` + `override` — динамический полиморфизм (вызов по реальному типу).
- `new` — **сокрытие** метода базового класса (вызов зависит от типа ссылки, не от реального типа) — источник багов.
- `sealed` — запрет дальнейшего переопределения/наследования; может помочь оптимизациям (devirtualization).

---

### Вопрос: Полиморфизм статический vs динамический
**Краткий ответ:**
- **Статический** — разрешается на компиляции: перегрузка методов (overloading), дженерики.
- **Динамический** — на рантайме: виртуальные методы/интерфейсы (overriding), `dynamic`.
