# 25. Soft skills и поведенческие вопросы

<a id="top"></a>

> Статус: 🟡 в работе · Уровень: Senior

## Что проверяют
Senior — это не только код: лидерство, коммуникация, принятие решений, работа с конфликтами и наставничество.

## Метод ответа: STAR
- **S**ituation — контекст
- **T**ask — задача/проблема
- **A**ction — что ты сделал (фокус здесь, от первого лица)
- **R**esult — результат (желательно в цифрах/фактах) + чему научился

## Рекомендации к ответам

<a id="q1"></a>
### Сложная техническая проблема
Опиши проблему, гипотезы, как **диагностировал** (метрики/профайлер/логи — связь с темами 20, 22), решение и результат. Покажи системность, а не «угадал».

<a id="q2"></a>
### Конфликт в команде
Фокус на **разрешении**, не на «кто виноват»: выслушал, нашёл общие цели, предложил решение/компромисс, договорились. Показать эмпатию и зрелость.

<a id="q3"></a>
### Несогласие с решением
«Disagree and commit»: аргументировал данными/прототипом, выслушал контекст, если решение осталось — конструктивно поддержал. Не «продавил» и не «саботировал».

<a id="q4"></a>
### Ошибка в проде
Без перекладывания вины: что произошло, как **быстро локализовал и починил** (rollback/hotfix), что сделал, чтобы **не повторилось** (тесты, мониторинг, постмортем без поиска виноватых). Ценится культура blameless.

<a id="q5"></a>
### Архитектурное решение
Контекст и ограничения → рассмотренные альтернативы → **trade-offs** → выбор и почему → как валидировал. Связь с темами 7, 15, 23. Упомяни ADR.

<a id="q6"></a>
### Менторство
Конкретный пример: как помог вырасти джуну (парное программирование, ревью, постановка задач по уровню), результат для человека и команды.

<a id="q7"></a>
### Жёсткие сроки / неопределённость
Декомпозиция, приоритизация (MVP, что можно отрезать), коммуникация ожиданий со стейкхолдерами, управление рисками. Не «героизм ценой качества».

<a id="q8"></a>
### Объяснение не-техническим людям
Язык бизнес-ценности/аналогии, без жаргона; связать техническое решение с влиянием на продукт/деньги/сроки.

<a id="q9"></a>
### Code review
**Давать:** по существу, про код а не человека, предлагать, а не приказывать, отмечать хорошее, разделять блокеры и nitpick. **Принимать:** без эго, благодарить, обсуждать, а не защищаться.

<a id="q10"></a>
### Приоритизация и техдолг
Как балансируешь фичи и техдолг: делать видимым (метрики, риски), встраивать в планирование, чинить по ходу (boy scout rule), аргументировать бизнес-языком.

<a id="q11"></a>
### Провал / достижение
Честно: достижение — с влиянием и твоей ролью; провал — с извлечённым уроком и изменением поведения.

<a id="q12"></a>
### Почему уходишь / почему к нам
Без негатива о текущем работодателе; про рост/интересные задачи/технологии; показать, что изучил компанию.

## Вопросы работодателю (важно задать)
- [ ] Процессы: code review, тестирование, релизы
- [ ] Технический долг — как с ним работают
- [ ] On-call, инциденты, постмортемы (культура blameless?)
- [ ] Рост, карьерный трек для Senior+
- [ ] Состав/зрелость команды, кто принимает технические решения
