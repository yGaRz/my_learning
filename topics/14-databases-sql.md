## SQL и оптимизация

### Индексы
**Краткий ответ:**
- **Clustered** — определяет физический порядок строк в таблице (обычно по PK). Один на таблицу.
- **Non-clustered** — отдельная структура (B-tree) со ссылками на строки. Много на таблицу.
- **Составной (composite)** — по нескольким колонкам; важен порядок колонок (левосторонний префикс).
- **Covering (покрывающий)** — содержит все колонки запроса (через ключ + `INCLUDE`), позволяет ответить без обращения к таблице.

### Когда индекс не используется, селективность
**Краткий ответ:** Индекс не помогает при низкой селективности (мало уникальных значений), функциях над колонкой (`WHERE YEAR(date)=...`), неявном приведении типов, `LIKE '%x'` (ведущий wildcard), нарушении префикса составного индекса. Селективность = доля уникальных значений; высокая → индекс эффективнее.

### План выполнения, seek vs scan
**Краткий ответ:** План показывает, как СУБД выполняет запрос. **Index Seek** — точечный поиск по индексу (хорошо). **Scan** (table/index) — проход всего (часто плохо для больших таблиц). Анализируют план для поиска bottleneck'ов и недостающих индексов.

### JOIN, подзапросы, EXISTS vs IN
**Краткий ответ:** Типы JOIN (inner/left/right/full/cross). `EXISTS` часто эффективнее `IN` с подзапросом (останавливается на первом совпадении, лучше с NULL). Коррелированные подзапросы могут быть дорогими.

### Оконные функции, CTE
**Краткий ответ:** Window functions (`ROW_NUMBER()`, `RANK()`, `SUM() OVER(...)`) — агрегаты без схлопывания строк. CTE (`WITH`) — именованные подзапросы для читаемости и рекурсии.

### Нормализация/денормализация
**Краткий ответ:** Нормализация (1NF–3NF) убирает дублирование и аномалии обновления. Денормализация осознанно добавляет избыточность ради скорости чтения (меньше JOIN) — trade-off целостность vs производительность.

## Транзакции и конкурентность

### ACID
**Atomicity** (всё или ничего), **Consistency** (инварианты сохраняются), **Isolation** (транзакции не мешают друг другу), **Durability** (зафиксированное переживёт сбой).

### Уровни изоляции и аномалии
| Уровень | Dirty read | Non-repeatable read | Phantom |
|---------|-----------|---------------------|---------|
| Read Uncommitted | возможен | возможен | возможен |
| Read Committed | нет | возможен | возможен |
| Repeatable Read | нет | нет | возможен |
| Serializable | нет | нет | нет |
| Snapshot | нет | нет | нет (через версии) |

- **Dirty read** — чтение незакоммиченных данных.
- **Non-repeatable read** — повторное чтение строки даёт другое значение.
- **Phantom** — повторный диапазонный запрос возвращает новые строки.

Выше изоляция → меньше аномалий, но больше блокировок/меньше конкурентность.

### Блокировки, deadlock
**Краткий ответ:** Блокировки (shared/exclusive) обеспечивают изоляцию. **Deadlock** — две транзакции взаимно ждут ресурсы друг друга; СУБД выбирает «жертву» и откатывает. Профилактика: единый порядок доступа к ресурсам, короткие транзакции, нужные индексы.

### Optimistic vs pessimistic locking
**Краткий ответ:**
- **Pessimistic** — блокировать строку на время работы (`SELECT ... FOR UPDATE`). Гарантия, но снижает конкурентность.
- **Optimistic** — не блокировать, проверять версию при сохранении (см. [тему 13](./13-ef-core-data-access.md)). Лучше при редких конфликтах.

## Масштабирование

### Репликация
**Краткий ответ:** Master (write) + read-replicas (чтение) масштабируют чтение и дают отказоустойчивость. Реплики обычно асинхронны → лаг репликации (read-your-writes проблема).

### Шардирование и партиционирование
**Краткий ответ:**
- **Партиционирование** — разбиение таблицы на части (по диапазону/хешу/списку) в пределах одной БД.
- **Шардирование** — распределение данных по разным серверам/БД (горизонтальное масштабирование записи). Сложности: кросс-шард запросы, выбор ключа шардирования, ребалансировка.

### CAP
**Краткий ответ:** При сетевом разделении (**P**) система выбирает между **C** (consistency) и **A** (availability). CP — отказ в обслуживании ради консистентности; AP — доступность ценой устаревших данных. PACELC дополняет: даже без разделения — выбор latency vs consistency.

## NoSQL

### Типы и выбор
- **Документные** (MongoDB) — гибкая схема, агрегаты-документы.
- **Key-value** (Redis) — простой и быстрый доступ по ключу.
- **Колоночные** (Cassandra) — большие объёмы, запись-heavy, временные ряды.
- **Графовые** (Neo4j) — связи/обходы графа.

**SQL vs NoSQL:** SQL — строгая схема, транзакции, сложные запросы, реляционные данные. NoSQL — масштабирование, гибкая схема, специфичные паттерны доступа. Часто используют вместе (polyglot persistence).

### BASE vs ACID
**Краткий ответ:** Многие NoSQL следуют **BASE** (Basically Available, Soft state, Eventual consistency) — доступность и масштаб ценой строгой согласованности, в отличие от ACID реляционных БД.
